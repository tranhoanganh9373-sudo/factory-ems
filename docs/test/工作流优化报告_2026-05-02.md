## 2. 浪费分析

### 2.1 TIMWOOD 七种浪费映射

| 浪费类型 | 实例 | 频次 | 耗时 | 占比 |
|---------|------|------|------|------|
| **等待** (Waiting) | Token 过期 ×2 → 重新认证 | 2次 | ~8min | 5.7% |
| **缺陷** (Defects) | k6 脚本 response 格式不匹配 (auth-login/dashboard/report-monthly) | 3个 | ~5min | 3.6% |
| **过度处理** (Over-processing) | Evidence collector 重跑 API tester 已验证的端点 | 12+ | ~6min | 4.3% |
| **移动/切换** (Motion) | 每轮重新发现 API 路径 (404×4轮) | 4轮 | ~12min | 8.6% |
| **库存** (Inventory) | 5个k6脚本中3个不可用，占据认知空间 | 3/5 | ~2min | 1.4% |
| **过度生产** (Over-production) | 4份报告重复包含环境/版本/仪表清单 | 4处 | ~4min | 2.9% |
| **未利用人才** (Skills) | 6角色手工逐个切换，无并行流水线 | 全流程 | ~0 | 0% |

**总浪费：~37min (26.4%)**

### 2.2 根因分析

```
Token过期根因链:
  JWT 900s (15min) ← BCrypt登录238ms ← 每轮需单独登录
  → 测试轮次跨15min时Token必然过期
  → 根本原因: 无Token刷新自动化机制

API路径重复发现根因链:
  无共享API路由清单 ← 前端API文件未在测试前解析
  → 每轮靠curl试探 → 404 → 翻前端源码 → 再试
  → 根本原因: 无预处理步骤提取API路由

k6脚本不可用根因链:
  response格式{data:{data:{...}}} ← 前端axios interceptor剥离外层data
  → k6直接调API拿到双层嵌套 → check失败
  → 根本原因: 无k6→实际API的对齐验证步骤

Evidence重跑根因链:
  前轮测试结果未持久化为结构化数据
  → 后轮无法直接引用 → 必须重新调用API获取
  → 根本原因: 测试产物非机器可读
```

## 5. CI/CD 流水线设计

```yaml
# .github/workflows/quality-gate.yml (新文件)
name: Quality Gate
on:
  pull_request:
    paths: ['ems-*/**', 'frontend/src/api/**']
  workflow_dispatch:  # 手动触发全流程

jobs:
  phase0-preflight:
    runs-on: ubuntu-latest
    outputs:
      routes: ${{ steps.extract.outputs.routes }}
    steps:
      - uses: actions/checkout@v4
      - name: Extract API routes
        id: extract
        run: |
          grep -roh 'apiClient\.\(get\|post\|put\|delete\)([^)]*' frontend/src/api/*.ts \
            | sed 's/apiClient\.//' | sort -u > api-routes.json
          cat api-routes.json

  phase1-automated:
    needs: phase0-preflight
    runs-on: ubuntu-latest
    steps:
      - name: Health check
        run: curl -f http://factory-ems:8888/actuator/health
      - name: k6 smoke tests
        uses: grafana/k6-action@v0.3
        with:
          filename: perf/k6/mixed-load.js
          flags: --duration 30s --vus 5

  phase2-parallel:
    needs: phase1-automated
    strategy:
      matrix:
        role: [api-tester, reality-checker, evidence-collector]
    runs-on: ubuntu-latest
    steps:
      - name: Run ${{ matrix.role }}
        run: ./scripts/test-pipeline.sh ${{ matrix.role }}

  phase3-analysis:
    needs: phase2-parallel
    runs-on: ubuntu-latest
    steps:
      - name: Aggregate results
        run: python3 scripts/aggregate-results.py --output final-report.md
      - uses: actions/upload-artifact@v4
        with:
          name: quality-report
          path: final-report.md
```
