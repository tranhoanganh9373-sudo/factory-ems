## 1. v1.1.5 — Track B 拓扑一致性自动报警

### 1.1 目标 (一句话)

把现有手动 `/alarms/health` 页面的 `topologyConsistency()` 检查变成**每小时自动跑一次**的后台任务，**严重 (>15%) 残差自动写入 alarms 表并推送通知**，回到健康区间自动 ack，并保留小时级趋势历史。

### 1.2 整体架构

```
┌──────────────────────────────────────────────────────────────────┐
│                  ems-dashboard (现有模块)                        │
│  DashboardServiceImpl.topologyConsistency(query)  ── 已有        │
└─────────────────────────────┬────────────────────────────────────┘
                              │ List<TopologyConsistencyDTO>
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│  ems-alarm 新增组件 (Track B 主体)                               │
│                                                                  │
│  TopologyConsistencyScheduler  ──┐                               │
│    @Scheduled(cron = hourly)     │                               │
│                                  ▼                               │
│  TopologyConsistencyAlarmService                                 │
│    ├─ for each row:                                              │
│    │    apply hysteresis state machine                           │
│    │    if ENTER: alarm + (history row + webhook fan-out)        │
│    │    if SUSTAIN: bump last_seen_at                            │
│    │    if EXIT: auto-ack + history row                          │
│    └─ persist:                                                   │
│         alarms (existing)                                        │
│         topology_consistency_history (NEW)                       │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                现有 AlarmEventBus / WebhookSender
                (alarm-bridge-recipes.md: feishu/email/sms)
```

### 1.3 数据模型 — 新表

#### `topology_consistency_history` (NEW migration V1_1_5_0)

每小时×每对父子表写一行。无 PK，仅 (parent_meter_id, sampled_at) 唯一索引；7 天 TTL（Postgres pg_partman 或简单定时清理 cron）。

```sql
CREATE TABLE topology_consistency_history (
    id BIGSERIAL PRIMARY KEY,
    parent_meter_id BIGINT NOT NULL,
    energy_type VARCHAR(16) NOT NULL,
    parent_reading NUMERIC(20,4) NOT NULL,
    children_sum NUMERIC(20,4) NOT NULL,
    children_count INT NOT NULL,
    residual NUMERIC(20,4) NOT NULL,
    residual_ratio NUMERIC(8,5),  -- nullable when parent_reading=0
    severity VARCHAR(16) NOT NULL, -- OK / INFO / WARN / WARN_NEGATIVE / ALARM
    sampled_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX uq_topo_hist_parent_sampled
    ON topology_consistency_history (parent_meter_id, sampled_at);
CREATE INDEX ix_topo_hist_sampled ON topology_consistency_history (sampled_at);
```

#### `alarms` 表复用（无 schema 变更）

新增 AlarmType 枚举值 `TOPOLOGY_NEGATIVE_RESIDUAL`，severity = MEDIUM。`device_id` 写 parent_meter_id，`device_type` = `METER`。`detail JSONB` 携带：

```json
{
  "parent_code": "MAIN-A",
  "parent_name": "厂区主进线",
  "energy_type": "ELEC",
  "residual_ratio": -0.184,
  "children_count": 3,
  "trigger_threshold": -0.15,
  "exit_threshold": -0.10,
  "first_seen_at": "2026-05-04T08:00:00Z"
}
```

### 1.4 滞后状态机 (hysteresis)

**核心目的**：避免边缘抖动（-15% 上下浮动）一个小时一条报警。

```
                         ratio ≤ -15%                 ratio ≥ -10%
       ┌───────────┐  ─────────────────► ┌───────────┐ ─────────► ┌───────────┐
       │   QUIET    │                    │  ALARMING │            │  QUIET    │
       │ (no row)   │ ◄──────────────── │  (active) │ ─────────► │ + auto-ack│
       └───────────┘    ratio > -10%     └───────────┘            └───────────┘
                       (QUIET stays QUIET)  ratio still ≤ -10%
                                            (ALARMING stays
                                             ALARMING, bump
                                             last_seen_at)
```

**实现**：Scheduler 每跑一次先用 `parent_meter_id` 查 alarms 表是否有 ACTIVE 的 `TOPOLOGY_NEGATIVE_RESIDUAL`：
- **无 + ratio ≤ -0.15** → 新建 alarm + 推送 webhook
- **无 + -0.15 < ratio ≤ -0.10** → 不动（仍在 QUIET）
- **有 + ratio ≤ -0.10** → 仅 update `last_seen_at`、追加 history row
- **有 + ratio > -0.10** → 自动 ack：set status=ACKED, acked_at=now(), detail += `{auto_ack_reason:"residual_recovered", auto_ack_at:...}` + 推送恢复通知

### 1.5 通知渠道

复用现有 `alarm-bridge-recipes.md` 的 webhook fan-out（feishu / email / sms）。MEDIUM 也推（与 Q2 决策一致）。模板新增：

- 触发：`【数据质量】{parent_name} 子表合计与父表偏差 -{ratio}%（>15% 阈值），可能 CT 倍率错 / 拓扑错配。详情 → /alarms/health`
- 恢复：`【数据质量】{parent_name} 拓扑残差已恢复至 {ratio}%，自动 ack。`

### 1.6 错误处理

- **Scheduler 跑挂**：吞异常 + log.error + Micrometer counter `topology_alarm_scheduler_failures_total`，下一小时重新跑一次（无补偿，因为是周期性）。
- **History 表写满**：日清理 cron 删除 `sampled_at < NOW() - 7 days` 的行。
- **Webhook 推送失败**：现有 fan-out 已有重试 + 死信队列，复用即可。

## 2. v1.2.0 — Track A 光伏接入

### 2.1 目标 (一句话)

允许工厂同时接入**商业电网 + 光伏（带上网）**，正确计算**总用电量、自用率、上网卖电收入抵扣、CO₂ 当量减排**，并在面板呈现「来源构成 GRID vs SOLAR」「光伏发电 vs 厂区用电曲线」「节费 / 碳排报表」。

### 2.2 核心公式

| 量 | 公式 |
|----|------|
| 厂区总用电 | `Σ(GRID_TIE.import) − Σ(GRID_TIE.export) + Σ(GENERATE.production)` |
| 自发自用（小时） | `max(0, hourly_pv_generation − hourly_grid_export)` |
| 自用率 | `Σ(self_consumption_hourly) / Σ(pv_generation_hourly)` |
| 上网电量 | `Σ(GRID_TIE.export)` |
| 上网卖电收入 | `Σ_period(grid_export_period × feed_in_tariff_period)` |
| 月度电费净额 | 现有 cost.monthly_bill − 上网卖电收入 |
| 减排 CO₂ | `Σ(self_consumption × carbon_factor[GRID − SOLAR])` |

### 2.3 数据模型 — 新字段 + 新表 (V2_5_0)

```sql
-- meter 三个新字段
ALTER TABLE meter
    ADD COLUMN role VARCHAR(16) NOT NULL DEFAULT 'CONSUME',
        -- CONSUME (默认，纯耗电) / GENERATE (光伏/风电产出表) / GRID_TIE (并网点，可双向)
    ADD COLUMN energy_source VARCHAR(16) NOT NULL DEFAULT 'GRID',
        -- GRID / SOLAR / WIND / STORAGE
    ADD COLUMN flow_direction VARCHAR(16) NOT NULL DEFAULT 'IMPORT';
        -- IMPORT / EXPORT / BIDIRECTIONAL
ALTER TABLE channel_point
    ADD COLUMN direction VARCHAR(8);
        -- NULL / IMPORT / EXPORT — 仅对 BIDIRECTIONAL meter 才需指定

-- 碳排因子（按区域 + 能源类型 + 生效日）
CREATE TABLE carbon_factor (
    id BIGSERIAL PRIMARY KEY,
    region VARCHAR(64) NOT NULL,
    energy_source VARCHAR(16) NOT NULL,
    effective_from DATE NOT NULL,
    factor_kg_per_kwh NUMERIC(10,4) NOT NULL,
    UNIQUE (region, energy_source, effective_from)
);

-- 上网电价（分时段）
CREATE TABLE feed_in_tariff (
    id BIGSERIAL PRIMARY KEY,
    region VARCHAR(64) NOT NULL,
    energy_source VARCHAR(16) NOT NULL,
    period_type VARCHAR(8) NOT NULL,  -- PEAK / FLAT / VALLEY / TIP
    effective_from DATE NOT NULL,
    price NUMERIC(10,4) NOT NULL,     -- ¥/kWh
    UNIQUE (region, energy_source, period_type, effective_from)
);
```

**默认值策略保后向兼容**：v1.1.x 已有 meter 全部默认 `role=CONSUME / energy_source=GRID / flow_direction=IMPORT`，行为完全不变。

### 2.4 拓扑分离

PV **不挂**在 GRID_TIE 根表下：

```
   原 v1.1.0 拓扑                  v1.2.0 with-pv 拓扑
   ┌──────────────┐               ┌──────────┐  ┌──────────┐
   │  MAIN (root) │               │ MAIN-GRID│  │  PV-ROOT │
   └──────┬───────┘               │ (GRID_TIE│  │ (GENERATE│
   ┌──────┼──────┐                │ BIDIRECT)│  │  EXPORT) │
   ▼      ▼      ▼                └─┬─────┬──┘  └────┬─────┘
  车间A 车间B 车间C                  ▼     ▼          │
                                   车间A 车间B   (并网到 MAIN-GRID
                                                  export 寄存器)
```

**两个根表**都参与「visible roots → KPI total」。`DashboardSupport.filterToVisibleRoots` 已具备多根表支持。

### 2.7 边界场景

| 场景 | 处理 |
|------|------|
| 客户没光伏（v1.1.x 老客户） | 默认值无变化；面板自动隐藏 PV 卡（PV-related 数据为 0 时隐藏） |
| 三相不平衡光伏 | 不在 v1.2.0；总线读数即可；后续按相处理 |
| 反孤岛 / 自发自用不上网 | 用 GRID_TIE export = 0 表达，无需新模型 |
| 单 BIDIRECTIONAL 表无 EXPORT 通道 | 拒绝创建，提示「双向表必须配置 IMPORT + EXPORT 双通道」 |
| 历史无 PV 的报表周期跨越 PV 上线日 | 按日 join：上线前 self_consumption=0、carbon_savings=0 |
