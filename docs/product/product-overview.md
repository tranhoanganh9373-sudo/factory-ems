# Factory EMS · 产品介绍

> 适用版本：v1.7.0+ ｜ 最近更新：2026-05-01
> 受众：销售 / 客户决策者 / 项目方 / 实施工程师
> 阅读时长：约 8 分钟

---

## §1 一句话

Factory EMS 是一套面向工业园区与制造工厂的能源管理一体化平台，把"采、看、算、管"四件事做在一个产品里。可独立部署，不必再自建监控基础设施或拼装多套开源工具。

适合的场景：已经在用 PLC / SCADA / 网关采集仪表数据，但缺少跨车间的统一视图，没有自动化报表与分摊，报警也没有形成闭环。

---

## §2 解决什么问题

- **数据散落在 PLC、SCADA、Excel 里，跨车间汇总靠人手**
  现场已经有仪表，但数据停在车间级 SCADA 或工程师电脑里。月底要对账、要交能源审计报告，靠人工抄表和 Excel 拼，慢、错、还无法支撑 KPI 决策。

- **断线无感知，等月底对账才发现数据缺了一大段**
  仪表掉电、网线松动、采集程序卡死——这些事件通常没人盯着。等月度报表生成、出现数据空洞，再回头补录已经晚了，影响碳核查、能效报告、对内绩效。

- **能耗成本要分摊到部门 / 产线，但规则复杂、人工算来不及**
  分摊规则会随组织调整、产线改造频繁变化（按面积、按人头、按产量、按平米单耗）。Excel 公式错一处，整张账单要重做，财务和运营各执一词。

- **电价分时 / 阶梯 / 季节复杂，账单计算靠老员工"经验"**
  尖峰平谷 + 阶梯 + 容量电费 + 力调电费混合在一起，工厂内部账单与电网账单经常对不上。换了人就维护不动，难以审计。

- **运维多套通知渠道，断线报警要在钉钉、企微、ITSM 之间来回搬**
  不同部门在不同系统看报警。每出一次异常都要广播一次、记录一次，容易遗漏。已有的报警系统又不带能源场景的业务语义（设备级阈值、维护模式）。

- **客户要看自己的能耗数据，但权限粒度难管**
  内部协作 + 外部租户 + 审计员，每个人能看到的组织节点不一样。统一登录、按节点授权、审计追溯，自己拼一套要花半年。

## §3 核心能力

- **多协议采集 + 缓冲落盘 + 断线续传**：内置 OPC UA / Modbus TCP / MQTT 三种主流协议，断网期间数据落本地 SQLite，恢复后自动回灌，避免数据丢失。
- **时序数据双库（PostgreSQL + InfluxDB）**：原始读数进 InfluxDB（高写入、按时间查询），业务元数据进 PostgreSQL（事务、关联、审计），各取所长。
- **组织树 + 仪表树**：厂区 → 车间 → 产线 → 设备的层级结构，支持权限按节点下发，报表按节点聚合，账单按节点拆分。
- **班次 / 产线 / 产量录入**：把生产维度数据接入能源平台，支撑"单位产量能耗"、"班次能耗对比"等运营 KPI。
- **分时电价 + 阶梯电价 + 容量电费**：复杂电价规则在系统内沉淀，账单可按月、按节点自动出。
- **成本分摊引擎**：规则化配置（按面积 / 按人头 / 按产量 / 自定义比例），异步运行、留全套审计明细。
- **日 / 月 / 年 / 班次报表**：开箱即用的报表模板，支持导出 Excel、CSV，可以做能耗趋势、Top 设备、班次对比等。
- **平面图可视化**：把仪表挂到 CAD / 平面图上，实时看每个点的状态与读数，运维和参观演示一目了然。
- **采集中断报警 + 维护模式 + Webhook 外发**：质量哨兵，断线第一时间通知，维护期可静默，外发钉钉 / 企微 / ITSM 用一个 Webhook 地址搞定。
- **可观测性栈（自带 SLO / 监控 / 看板）**：内置 17 个业务指标 + 4 个 SLO + 7 个 Grafana 看板，运维上线即可监控平台自身健康。
- **细粒度 RBAC + 节点级权限 + 全程审计**：4 个角色（管理员 / 运维 / 财务 / 查看者）+ 节点级数据隔离 + 全部敏感操作进审计日志。
- **一键 Docker 部署**：单 compose 文件起全栈（Postgres + InfluxDB + 后端 + 前端 + nginx），现场最短 30 分钟可跑通。

## §4 适合谁

### 客户类型 A：制造工厂 / 工业园区

**典型规模**：年用电 500 万 kWh 以上，2~10 个车间，仪表数 50~500 块。
**关注点**：能源审计合规、单位产量能耗 KPI、跨车间能耗对比、内部分摊到产线 / 部门。
**为什么选 Factory EMS**：业务语义贴近能源管理，开箱即用的电价 + 报表 + 分摊，不用自己拼装。

### 客户类型 B：能源服务公司（ESCO）/ 节能改造项目方

**典型规模**：服务 5~50 个客户场地，需要交付月度能源报告和数据完整性证明。
**关注点**：数据采集完整率（碳核查通常要求 ≥ 95%）、报告自动化、多客户场地隔离。
**为什么选 Factory EMS**：采集中断报警 + 节点级权限隔离 + 自动报表，一套平台服务多个项目。

### 客户类型 C：园区运营方 / 物业管理公司

**典型规模**：1~5 栋楼，租户数 10~100，按租户独立计量与计费。
**关注点**：租户独立账单、电价分时透明、租户能看自己的数据但看不到别家。
**为什么选 Factory EMS**：内置账单引擎 + 租户角色 + 审计追溯，对账与发票流程清晰。

### 客户类型 D：碳核查 / 双控合规 / 重点用能企业

**典型规模**：受双控政策约束的钢铁、化工、建材、电子等行业。
**关注点**：数据完整可追溯、按工序 / 产品聚合能耗、应对突击核查。
**为什么选 Factory EMS**：审计日志 + 时序数据保真 + 报表导出，能直接对接核查机构的取数要求。

## §5 不是什么（划界）

以下能力不在 Factory EMS 范围内，避免与其他系统功能重叠或预期错配：

- **不是 SCADA / DCS / PLC 替代品**：Factory EMS 不直接控制现场设备，不下发开关命令、不做安全联锁。它从已有 SCADA / 网关 / PLC **读数据**，不替代它们的控制功能。
- **不是 MES / ERP**：不管理工单、物料、库存、订单。生产维度只录入班次和产量两个字段，用于能耗-产量关联分析。
- **不是通用 BI 平台**：报表模板针对能源场景预设，不提供完全自由的拖拽式数据透视。如需自由分析，通过导出 CSV 或直连 InfluxDB / PostgreSQL 接入第三方 BI。
- **不是碳核查认证机构**：平台输出符合核查要求的数据与报告，但不替代第三方机构的合规审核。
- **不内置短信 / 邮件 / IM 直连**：报警仅提供 Webhook，钉钉 / 企业微信 / 短信网关请通过 Webhook 桥接。
- **不内置多租户 SaaS 化运营**：当前版本面向单实例独立部署，不内置多租户计费、订阅、白标等运营能力。
- **不是 EMS 标准协议网关**：不实现 IEC 60870-5-104、DLT 645 等电力行业专用协议（如有需求请定制开发或通过中间网关转换）。

## §6 平台架构总览

```
                          ┌─────────────────────────────────┐
                          │         前端（React + AntD）      │
                          │  仪表盘 / 报表 / 平面图 / 配置    │
                          └────────────┬────────────────────┘
                                       │ HTTPS (nginx)
                          ┌────────────▼────────────────────┐
                          │      后端（Spring Boot · Java 21） │
                          │                                  │
                          │  ┌─────────────────────────────┐ │
                          │  │ 业务层                        │ │
                          │  │ orgtree / meter / tariff /   │ │
                          │  │ production / floorplan /     │ │
                          │  │ cost / billing / report /    │ │
                          │  │ dashboard / alarm            │ │
                          │  └─────────────────────────────┘ │
                          │  ┌─────────────────────────────┐ │
                          │  │ 平台层                        │ │
                          │  │ auth · audit · observability │ │
                          │  └─────────────────────────────┘ │
                          │  ┌─────────────────────────────┐ │
                          │  │ 采集层                        │ │
                          │  │ collector (OPC UA/Modbus/MQTT)│ │
                          │  └─────────────────────────────┘ │
                          └──┬───────────────┬───────────────┘
                             │               │
              ┌──────────────▼──┐      ┌─────▼─────────────┐
              │   PostgreSQL    │      │     InfluxDB      │
              │ (业务数据 + 元数据) │      │  (时序读数 · 高写入) │
              └─────────────────┘      └───────────────────┘
                     ▲
                     │ 读
              ┌──────┴──────────┐
              │  现场 PLC / SCADA / │
              │  网关 / 智能仪表   │
              └─────────────────┘
```

- **前端**：单页应用，AntD 组件库，所有功能通过浏览器访问。
- **后端**：模块化 Spring Boot 单体（16 个 Maven 模块），统一 JWT 鉴权、事务、审计。
- **采集**：独立线程池 + 缓冲落盘 + 断线续传，可配置开关。
- **存储**：PostgreSQL 跑业务事务（迁移由 Flyway 自动执行），InfluxDB 跑时序数据。
- **部署**：Docker Compose 一键起栈，单实例足够支撑 500 块仪表 / 10 万条/天的规模。

## §7 同类竞品对比

| 对比维度 | Factory EMS | 自建（开源拼装）| 第三方 SaaS（按月付费）| SCADA 自带能源模块 |
|---|---|---|---|---|
| 部署方式 | 私有部署，单 compose 起栈 | 自维护 N 个组件 | 数据上云，无需部署 | 与 SCADA 强绑定 |
| 业务语义 | 能源场景预设（电价 / 分摊 / 班次 / 报表） | 通用监控，需大量自定义 | 因厂商而异，多面向租户 | 偏控制，能源弱 |
| 多协议采集 | OPC UA / Modbus / MQTT 内置 | 需选型 + 集成 | 多通过网关上传 | 多与控制系统耦合 |
| 报表与电价 | 分时 / 阶梯 / 容量内置 | 需写代码 | 通用模板，本地化弱 | 简单累计为主 |
| 成本分摊 | 规则引擎，按面积 / 人头 / 产量 | 通常没有 | 部分有，复杂规则受限 | 通常没有 |
| 数据私有 | 全部本地，符合涉密 / 内网 | 全部本地 | 数据上第三方云 | 全部本地 |
| 上手成本 | 30 分钟跑通；语义上手 1~2 周 | 多人月起 | 立即可用 | 学习门槛高 |
| 可定制性 | 模块化代码可二次开发 | 完全自由 | 受限于平台 | 受限于厂商 SDK |
| 长期成本 | 一次性部署 + 自维护 | 高维护成本 | 持续订阅 | 与 SCADA 升级绑定 |

差异点：电价、分摊、班次、维护模式、节点权限这些能源场景常见的功能，部署完即用，不需要客户自己拼装；数据全部留在本地，符合制造业对数据资产的私有化要求。

## §8 路线图（公开版）

> 已发布，可立即使用：

- **v1.1**：基础认证 / 组织树 / 仪表 / 审计
- **v1.2**：时序数据底座（InfluxDB）
- **v1.3**：报表 + 电价 + 班次 + 平面图
- **v1.5**：采集层（OPC UA / Modbus / MQTT）+ 断线缓冲
- **v1.6**：采集中断报警 + Webhook 外发
- **v1.7**：可观测性栈（17 指标 + 4 SLO + 7 Dashboard）
- **v2.0**：成本分摊引擎
- **v2.1**：账单引擎
- **v2.2**：报警增强（覆盖更多场景）
- **v2.3**：协议层与通道扩展

> 规划中，受版本节奏与客户需求影响：

- 多 Webhook 端点 + 报警严重程度分级
- IM 直连（钉钉 / 企微）
- 用户级订阅过滤
- 多实例集群部署
- IEC 60870-5-104 / DLT 645 等电力专用协议
- 多租户 SaaS 化（外部化运营）

具体版本节奏以项目方确认为准，以上仅为产品方向，不构成承诺。
