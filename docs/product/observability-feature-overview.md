# 可观测性栈 · 功能概览

> 适用版本：v1.7.0+ ｜ 最近更新：2026-04-29

---

## §1 一句话价值

把"线上跑得怎么样"从黑箱变成图表：性能数据、日志、请求轨迹在一处查，4 个 SLO 加 16 条告警把稳定性变成可量化的数字。

工程师不再 SSH 上服务器 grep 日志，客户也能直接看到"上个月可用性 99.62%"这种数据，用来支撑合同 SLA。
基础设施故障在客户察觉前先告警，远程支持成本随之下降，合同谈判和续约也更有数据可依。

---

## §2 解决什么问题

- **看不见系统状态，出问题靠猜**
  没有趋势图、没有 SLO、没有跨模块视角，每次异常都得 SSH 上服务器手动 grep 日志。
  恢复慢，事后也讲不清根因，复盘报告只能靠经验拼凑，下次同类问题还是从头再来。

- **故障被客户先发现，工程师一直在救火**
  没有自动告警，曲线断了才被反馈，团队长期处在"客户报障 → 远程排查"的状态里。
  周末半夜被电话叫醒，白天又排不上正常迭代节奏，工程师疲劳累积，人员流失风险也跟着上升。

- **客户问"系统稳不稳定？"答不上来**
  合同里写的可用性、响应时间没有数据支撑，谈续约、谈扩容、谈合规审计时只能靠主观印象搪塞。
  客户内部 IT 部门把关、要求月度 SLA 报告时拿不出东西，会影响后续的商务谈判和项目验收。

- **远程支持成本高，小故障也得派人飞过去**
  装到客户场地后，没有给客户运维看的标准 dashboard，小故障也要派工程师上门。
  部署越多越亏，客户运维想自查也无从下手，团队疲于跨城出差，难以规模化复制。

## §3 核心功能

- **3 信号统一收口**：metrics（性能数据）、logs（日志）、traces（请求轨迹）汇总到同一个 Grafana 界面。
  从某条告警一键跳到对应日志，再跳到对应请求链路，排障不用切换工具，不用再开 5 个浏览器标签来回比对。

- **17 项业务指标开箱即用**：覆盖采集、告警、计量等核心模块的入库速率、延迟、失败次数等。
  与 Spring Boot 自带的 JVM / HTTP 指标互补，给容量评估和性能优化提供数据基线，不再靠拍脑袋估算。

- **4 个 SLO + 16 条告警规则**：内置可用性 99.5%、API 响应延迟、数据新鲜度、调度漂移四项目标。
  告警分"严重 / 警告"两档，到阈值自动通知，不依赖人工盯盘；同时记录错误预算消耗，方便对客户承诺数字。

- **多通道告警分发**：邮件、钉钉、企业微信、自定义 Webhook 四种渠道按需启用。
  配置文件改完重启即可，可以把告警接入既有的运维通知体系（如内部值班群、ITSM 工单系统），不用另开告警通道、不用维护告警中间件。

- **7 个预置 Dashboard**：SLO 总览、基础设施、JVM、HTTP，加上采集 / 告警 / 计量三个业务模块大盘。
  登录即可查看，客户运维不用懂查询语法也能看懂；每张大盘预设了关键阈值和时间窗口，方便日常巡检和月度报告截图。

- **零应用层改造，独立部署**：可观测性栈作为独立组件部署，与产品主栈生命周期解耦。
  观测栈挂了不影响业务，业务升级也不用停观测；以后客户要换成自有监控体系（如 Zabbix 或企业 ELK），整套替换即可，产品栈不用动。

## §4 适用场景

### 场景 A：工程值班远程排障 — 30 分钟内定位问题

凌晨 3 点客户反馈"曲线断了"。值班工程师打开 Grafana SLO 总览，看到"数据新鲜度"红了，下钻到 ems-meter 大盘，发现电力数据入库速率从每分钟 100 条掉到 0；点击 Top 设备表的某行，跳到对应日志页，看到 timeout 报错堆栈，再跳到请求链路视图，确认是某 IP 的采集通道卡住。整个流程不用 SSH、不用 grep、不用打电话叫人，MTTR 从过去的小时级压到 30 分钟内。第二天运维例会上工程师拿出"已修复并附完整时间线"的复盘，客户对系统响应的信心也跟着上来了。

### 场景 B：容量评估与合同谈判 — 用真实数据兜底

工厂打算从 200 台仪表扩到 500 台。打开 SLO 总览看 30 天历史：可用性 99.62%、API p99 延迟 870 毫秒（均优于目标），但调度漂移在 200 台规模时已经接近 60 秒警戒线。结论：扩到 500 台前要先做调度优化，避免上线后翻车。给客户的合同里写"99.5% 可用性"有数据支撑，月度报告直接截图 dashboard 即可，不用再做 PPT 美化。客户内部 IT 审计或集团合规要月度 SLA 数据时，运维一键导出，原本可能拖几周的合规审查当天就能交。

### 场景 C：故障复盘与改进闭环 — 一屏还原事故现场

某次告警集中在凌晨 2 点。次日复盘时，运维负责人在 Grafana 圈定时间窗口，一屏看到 GC 暂停尖峰、JVM 堆内存涨到 90%、几条 webhook 失败连环出现的因果链。复盘报告直接截图三张大盘，工程团队两天内定位到是某个批量任务并发数过高。改完后 baseline 留在 dashboard 里，下次类似问题一眼就能看出"是不是又复发了"，事故改进从"模糊回忆"变成"数据对照"。

## §5 与 ems-alarm 的协同

可观测性栈和 ems-alarm 解决的是两个不同层面的告警问题，互补而不替代：

- **ems-alarm 在做的：单设备级业务告警**
  关注"某台仪表是不是还在正常采集数据"——采集中断、超时阈值、连续失败这类业务事件，是客户和运维直接关心的现象。
  告警发送到运维负责人和客户管理员手中，配套站内通知加 Webhook 闭环，每条告警都对应一台具体设备、一段具体的数据空洞。

- **可观测性栈在做的：基础设施级告警**
  关注"系统本身是不是健康"——应用是否宕机、JVM 是否承压、磁盘是否快满、错误率是否突增、调度是否漂移。
  这是工程师和系统层面的问题，发送到工程值班和系统运维，回答的是设备告警背后更深一层的"为什么 100 台设备同时掉线"。

- **两条通道分开，互不屏蔽**
  业务告警走 ems-alarm 自有的 Webhook 通道，基础设施告警走 Alertmanager 多通道分发，分组分级互不干扰。
  避免业务侧故障刷屏导致基础设施告警被淹没，反过来也一样——系统抖动一次也不会把所有客户的设备告警清空。

- **互补关系：业务告警驱动客户响应，基础设施告警驱动运维响应**
  客户运维收到 ems-alarm 后去现场看仪表、查线缆；工程团队收到可观测性栈告警后扩容、调优、修代码。
  两条线并行，分工清晰；当 ems-alarm 出现"100 台设备同时断线"这种模式时，工程师可以立刻查可观测性栈大盘，判断是应用宕机、调度堆积还是网络抖动。

简单说：ems-alarm 回答"客户的设备好不好"，可观测性栈回答"我们的系统好不好"。

实际部署时，两栈共享同一台服务器和同一个网络，但在 docker compose 层面相互独立：
- ems-alarm 跟着产品主栈一起发版升级，跟业务节奏；
- 可观测性栈作为独立组件部署，按运维节奏迭代，互不打扰。
两者都会出现在站内通知和客户运维例会中，责任边界清晰：业务异常找 ems-alarm，系统健康找可观测性栈。
