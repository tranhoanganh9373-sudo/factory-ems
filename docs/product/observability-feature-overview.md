# 可观测性栈 · 功能概览

> 适用版本：v1.7.0+ ｜ 最近更新：2026-04-29

---

## §1 一句话价值

把"线上跑得怎么样"从黑箱变成图表，从猜测变成证据：性能数据、日志、请求轨迹一站式查询，4 个 SLO + 16 条告警让稳定性有数字、有兜底、有闭环。

工程师不必再 SSH 进服务器 grep 日志，客户也能直接看到"上个月可用性 99.62%"这样的真实指标，给合同 SLA 兜底。
基础设施级故障在客户察觉前主动告警，远程支持成本下降、合同谈判与续约更有底气，团队从"被动救火"切换到"主动预警"，让能源管理系统真正具备 7×24 在线服务的工程能力。

---

## §2 解决什么问题

- **看不见系统状态，出问题靠猜**
  没有趋势图、没有 SLO、没有跨模块视角，每次异常都得 SSH 上服务器手动 grep 日志。
  恢复速度慢，事后也讲不清根因，复盘报告只能靠经验拼凑，下一次同类问题还是只能再来一遍。

- **故障被客户先发现，工程师永远在被动救火**
  没有自动告警通道，曲线断了才被反馈，团队一直处于"客户报障 → 远程排查"的被动状态。
  体验和口碑双输；周末半夜被电话叫醒、白天又排不上正常迭代节奏，工程师疲劳累积、人员流失风险升高。

- **客户问"系统稳不稳定？"答不上来**
  合同里写的可用性、响应时间没有真实数据支撑，谈续约、谈扩容、谈合规审计时一筹莫展。
  只能靠主观印象搪塞，遇到客户内部 IT 部门把把关、要求月度 SLA 报告时无从应对，影响后续商务谈判与项目验收。

- **远程支持成本高，每次问题都得现场上人**
  装到客户场地后，没有标准 dashboard 给客户运维看，每次小故障都得派工程师飞过去。
  部署越多越亏本；客户运维想自查也无从下手，团队疲于跨城出差，规模化复制根本无从谈起。

## §3 核心功能

- **3 信号统一收口**：metrics（性能数据）、logs（日志）、traces（请求轨迹）三类观测数据汇总到同一个 Grafana 界面。
  从某条告警一键跳到对应日志，再跳到对应请求链路，排障路径无需切换工具，工程师不必再开 5 个浏览器标签来回比对。

- **17 项业务指标开箱即用**：覆盖采集、告警、计量等核心模块的入库速率、延迟、失败次数等。
  与 Spring Boot 自带的 JVM / HTTP 指标互补，为容量评估和性能优化提供真实数据基线，告别拍脑袋估算。

- **4 个 SLO + 16 条告警规则**：内置可用性 99.5%、API 响应延迟、数据新鲜度、调度漂移四项核心目标。
  告警分级为"严重 / 警告"两档，到达阈值自动通知，不再依赖人工盯盘；同时记录"错误预算"消耗，让稳定性可量化、可对客户承诺。

- **多通道告警分发**：邮件、钉钉、企业微信、自定义 Webhook 四种渠道按需启用。
  配置文件改完重启即可，企业可以把告警接入既有的运维通知体系（如内部值班群、ITSM 工单系统），无需另开告警通道、无需维护告警中间件。

- **7 个预置 Dashboard**：SLO 总览、基础设施、JVM、HTTP，以及采集 / 告警 / 计量三个业务模块大盘。
  登录即可查看，客户运维不需要懂查询语法也能看懂；每张大盘均预设关键阈值与时间窗口，方便日常巡检和月度报告截图。

- **零应用层改造，独立部署**：可观测性栈作为独立组件部署，与产品主栈生命周期解耦。
  观测栈崩溃不影响业务，业务升级也不必停观测；日后客户想换成自有监控体系（如 Zabbix 或企业 ELK），可整套替换、产品栈零改动。

## §4 适用场景

### 场景 A：工程值班远程排障 — 30 分钟内定位问题

凌晨 3 点客户反馈"曲线断了"。值班工程师打开 Grafana SLO 总览，发现"数据新鲜度"红了，下钻到 ems-meter 大盘，看到电力数据入库速率从每分钟 100 条骤降到 0；点击 Top 设备表的某行，自动跳到对应日志页，看到 timeout 报错堆栈，再跳到请求链路视图，确认是某 IP 的采集通道卡住。整个流程不用 SSH、不用 grep、不用打电话叫人，平均故障定位时间（MTTR）从过去的小时级压缩到 30 分钟内。客户在第二天的运维例会上看到工程师"已修复并附完整时间线"的复盘，对系统响应能力的信心大幅提升。

### 场景 B：容量评估与合同谈判 — 用真实数据兜底

工厂打算从 200 台仪表扩到 500 台。打开 SLO 总览看 30 天历史，可用性 99.62%、API p99 延迟 870 毫秒（均优于目标），但调度漂移在 200 台规模时已接近 60 秒警戒线；据此得出结论：扩到 500 台前需要先做调度优化，避免上线后翻车。给客户的合同里写"99.5% 可用性"也有底气，月度报告直接截图 dashboard 即可，不需要再做 PPT 美化。客户内部 IT 审计或集团合规要求月度 SLA 数据时，运维一键导出，原本可能拖延数周的合规审查在当天就能交差。

### 场景 C：故障复盘与改进闭环 — 一屏还原事故现场

某次告警全集中在凌晨 2 点。次日复盘时，运维负责人在 Grafana 圈定时间窗口，一屏看到 GC 暂停尖峰、JVM 堆内存涨到 90%、多条 webhook 失败连环出现的因果链。复盘报告直接截图三张大盘，工程团队在两天内定位是某个批量任务并发数过高所致。改完后 baseline 留在 dashboard 里，下次类似问题一眼就能看出"是不是又复发了"，团队的故障改进闭环从"模糊回忆"变成"数据对照"，每次事故都成为系统稳定性提升的台阶。

## §5 与 ems-alarm 的协同

可观测性栈和 ems-alarm 解决的是**两个不同层面**的告警问题，互为补充而非替代：

- **ems-alarm 已经在做的：单设备级业务告警**
  关注"某台仪表是否还在正常采集数据"——采集中断、超时阈值、连续失败这类业务事件，是客户和运维直接关心的现象级问题。
  告警发送到运维负责人和客户管理员手中，配套站内通知 + Webhook 闭环，每一条告警都对应一台具体设备、一段具体的数据空洞。

- **可观测性栈在做的：基础设施级告警**
  关注"系统本身是否健康"——应用是否宕机、JVM 是否承压、磁盘是否快满、错误率是否突增、调度是否漂移。
  这是工程师和系统层面的问题，发送到工程值班和系统运维，是设备告警背后更深一层的"为什么 100 台设备同时掉线"的根因排查抓手。

- **两条通道分开，互不屏蔽**
  业务告警走 ems-alarm 自有的 Webhook 通道，基础设施告警走 Alertmanager 多通道分发，分组分级互不干扰。
  避免"业务侧故障刷屏导致基础设施告警被淹没"，反之亦然——系统抖动一次也不会把所有客户的设备告警全清空，每一类告警都得到对应专业角色的应有响应。

- **互补关系：业务告警驱动客户响应，基础设施告警驱动运维响应**
  客户运维收到 ems-alarm 后去现场看仪表、查线缆；工程团队收到可观测性栈告警后扩容、调优、修代码。
  两条线并行，事故响应分工清晰，避免相互甩锅；当 ems-alarm 出现"100 台设备同时断线"这种异常模式时，工程师可立即查看可观测性栈大盘，判断是否为应用宕机、调度堆积或网络抖动所致。

简言之：ems-alarm 回答"客户的设备好不好"，可观测性栈回答"我们的系统好不好"。
两个问题在能源管理场景里都不能少，缺一就会出现"客户在催设备、工程师没工具找原因"或"基础设施跑得好、客户却在投诉漏数据"的尴尬局面。

实际部署时，两栈共享同一台服务器与同一个网络，但在 docker compose 层面相互独立：
- ems-alarm 与产品主栈一同发版升级，跟随业务节奏；
- 可观测性栈作为独立组件部署，按运维节奏迭代，互不打扰。
两者都会出现在站内通知与客户运维例会中，但责任边界清晰：业务异常找 ems-alarm，系统健康找可观测性栈，分工明确、互不挤占。
