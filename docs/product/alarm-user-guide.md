# 采集中断报警 · 用户使用手册

**最近更新：** 2026-04-29
**适用版本：** v1.6.0+

本手册面向工厂运营人员（管理员和操作员），说明怎么用采集中断报警功能：查看报警、处理报警、配置规则，以及遇到常见问题怎么办。

---

## 1. 谁能做什么 — 角色对照表

系统按角色限制操作，下表说明各角色能做什么。如果发现自己没有某个操作权限，找管理员。

| 操作 | 管理员 (ADMIN) | 操作员 (OPERATOR) |
|------|:---:|:---:|
| 查看报警列表与详情 | ✅ | ✅ |
| 确认报警（ACTIVE → ACKED） | ✅ | ❌ |
| 手动恢复报警（→ RESOLVED） | ✅ | ❌ |
| 配置阈值 / 维护模式 | ✅ | ❌ |
| 配置 Webhook | ✅ | ❌ |
| 接收站内通知（铃铛） | ✅ | ✅ |

> **说明**：VIEWER 和 FINANCE 角色没有报警模块访问权限，铃铛图标对其隐藏。如果要查看报警，找管理员升级角色。

## 2. 操作员视角

### 2.1 看到铃铛上有红点该怎么办

顶部导航栏右上角的 🔔 铃铛图标带红色角标，显示当前活跃报警数量（只统计 ACTIVE 状态，每 30 秒自动刷新）。看到红点不要慌，按下面步骤处理：

**处理流程（5 步）：**

1. **查看角标数量**：铃铛角标数字就是当前 ACTIVE 报警总数。数字为 0 表示没有待处理报警。

   > 截图占位：顶栏右上角铃铛图标，带红色角标数字"3"

2. **点击铃铛打开通知抽屉**：展开最近 20 条 ACTIVE 报警，按触发时间降序排列，每条显示设备名称、报警类型和触发时间。

   > 截图占位：通知抽屉展开，列出 3 条 ACTIVE 报警，每条带红色"报警"标签

3. **判断优先级**：看"设备名称 / 报警类型 / 触发时间"。触发时间越早、设备越关键，优先级越高。报警类型说明：
   - 静默超时：设备超过设定时间没上报新数据
   - 连续失败：采集器对该设备连续采集失败多次

4. **确认或上报**：
   - 如果你是 ADMIN：直接在抽屉里点"确认"按钮，报警状态变为 ACKED（黄色），告诉其他同事"已有人认领"。
   - 如果你是 OPERATOR：点"查看详情"跳到报警历史页，把报警 ID 和设备信息告知管理员处理。

   > 截图占位：抽屉中单条报警的"确认"按钮（ADMIN 可见）与"查看详情"链接

5. **等待自动恢复**：设备数据恢复正常后，系统会在 5 分钟内自动把报警置为 RESOLVED（绿色），该条报警从铃铛抽屉消失，不用手动操作。

   > 截图占位：同一报警已变为绿色"已恢复"状态，从抽屉中消失

### 2.2 看报警历史

**路径：** 左侧菜单 → 系统健康 → 报警历史

报警历史页列出所有状态的报警记录，支持筛选和搜索。

**顶部筛选器：**

| 筛选项 | 可选值 |
|--------|--------|
| 状态 | ACTIVE（活跃）/ ACKED（已确认）/ RESOLVED（已恢复）/ 全部 |
| 类型 | 静默超时 / 连续失败 / 全部 |
| 时间范围 | 今日 / 近 7 天 / 近 30 天 / 自定义 |

**表格列含义：**

| 列名 | 含义 |
|------|------|
| 设备 | 触发报警的设备名称和 ID |
| 类型 | SILENT_TIMEOUT（静默超时）或 CONSECUTIVE_FAIL（连续失败） |
| 严重度 | 首版固定为 WARNING |
| 状态 | ACTIVE（红） / ACKED（黄） / RESOLVED（绿） |
| 触发时间 | 报警首次创建的时间 |
| 持续时长 | 从触发到当前或恢复的经过时间 |
| 操作 | 查看详情 / 确认（ADMIN）/ 手动恢复（ADMIN） |

**查看报警详情：** 点击任意行的"详情"按钮，弹出详情 Modal，内容包括：
- 完整时间线（触发 → 确认 → 恢复，每步的操作人和时间）
- `detail` 原始 JSON 字段（含触发时的设备状态快照）
- 恢复方式（AUTO 自动 / MANUAL 手动）

**URL 直达：** 在浏览器地址栏输入 `/alarms/history?id=123` 直接打开 ID 为 123 的报警详情，方便通过 IM 分享给同事。

### 2.3 看设备实时状态

查看单台设备的当前报警状态有两个入口：

**入口一：测点管理列表**

路径：左侧菜单 → 测点管理 → 设备列表

列表中的"报警"列有三种状态：
- 未启用：该设备没纳入报警监控（不在 collector 配置中）
- 报警中（红色徽章）：当前有 ACTIVE 或 ACKED 报警
- 正常（绿色徽章）：没有活跃报警

**入口二：仪表盘**

路径：左侧菜单 → 仪表盘

- 左上角"采集健康"卡：显示在线率百分比加当前报警总数。点击跳到健康总览页。
- Top N 图表：点击具体设备名称展开设备抽屉，顶部显示状态徽章，下方"最近报警"Timeline 列出该设备最近 5 条报警（颜色：红 = ACTIVE，黄 = ACKED，绿 = RESOLVED）。

## 3. 管理员视角

### 3.1 处理一条新报警

从报警触发到关闭的完整操作流程：

> 截图 1：铃铛角标从 0 变为 +1，红色角标出现

**第一步：发现报警**
铃铛角标出现或数字增加，说明有新的 ACTIVE 报警产生。系统同时推送站内通知，如果配了 Webhook 也会触发外发。

> 截图 2：通知抽屉打开，列出新的 ACTIVE 报警，红色"报警"标签，显示设备名和触发时间

**第二步：打开抽屉看情况**
点击铃铛打开通知抽屉，看是哪台设备、什么类型的报警、何时触发。判断是真实故障还是短暂抖动（短暂抖动通常 5 分钟后会自动恢复）。

**第三步：点"确认"认领报警**
确认后报警状态从 ACTIVE（红）变为 ACKED（黄），铃铛角标数字相应减少。这等于告诉所有人：此报警已有人跟进，不用重复响应。

> 截图 3：点击"确认"按钮前后对比，状态从红色 ACTIVE 变为黄色 ACKED

> **重要**：确认 ≠ 已解决。确认（ACKED）只表示有人在跟进，问题还没修复。系统会记录确认时间和确认人，便于审计。

**第四步：等设备恢复**
派维修人员或联系网络团队处理。设备数据恢复正常后，系统 5 分钟内自动把报警置为 RESOLVED。

> 截图 4：报警状态变为绿色"已恢复"，`resolved_reason = AUTO`，通知抽屉中该条消失

**第五步（可选）：手动关闭**
如果现场已确认设备修好但数据还没上报，或业务上不想再等，ADMIN 可以直接点"手动恢复"立即关闭报警，记录 `resolved_reason = MANUAL`。

**三态转换说明：**

```
ACTIVE（红） ──[ADMIN 点确认]──────────────────────────────→ ACKED（黄）
                                                              │
                                         [数据恢复 5min / 手动恢复]
                                                              ↓
ACTIVE（红） ──[数据恢复 5min]────────────────────────────→ RESOLVED（绿）
ACTIVE（红） ──[ADMIN 点手动恢复]─────────────────────────→ RESOLVED（绿）
```

RESOLVED 是终态，不能再转换。如果设备再次中断，系统会创建一条新的报警记录。

### 3.2 给某台设备单独配置阈值

**路径：** 系统健康 → 阈值规则 → 设备列表 → 找到目标设备 → 点击"编辑"

**全局默认值（只读）：**

页面顶部显示当前全局配置，这些值由后端 `alarm.detection.*` 参数控制，本页只能查看不能改。要调整全局值，需要由系统管理员改服务器配置。详情见 [alarm-config-reference.md](./alarm-config-reference.md)。

| 全局参数 | 默认值 | 含义 |
|---------|--------|------|
| 静默超时秒数 | 600 秒 | 超过 10 分钟无数据则触发 |
| 连续失败次数 | 3 次 | 连续采集失败 3 次则触发 |
| 抑制窗口秒数 | 300 秒 | 报警恢复后 5 分钟内不重复触发 |

**设备级覆盖（可编辑）：**

在设备列表中找到目标设备，点击"编辑"，可单独设置以下字段：

| 字段 | 说明 | 留空效果 |
|------|------|---------|
| 静默超时（秒） | 该设备专属的超时时长 | 留空 = 沿用全局默认 |
| 连续失败次数 | 该设备专属的失败阈值 | 留空 = 沿用全局默认 |
| 维护模式 | 开关，开启时跳过报警检测 | 默认关闭 |
| 维护备注 | 说明维护原因和计划时间 | 可选，建议填写 |

**生效时间：** 保存后立即生效，最长延迟一个调度周期（默认 60 秒）。下一轮检测就会使用新阈值。

**常用场景：**
- 高频采集设备（每 5 秒一次）：把静默超时调低到 60 秒，更快发现中断
- 网络不稳定设备：把连续失败次数调高到 10 次，减少抖动误报
- 低频采集设备（每小时一次）：把静默超时调高到 3600 秒，避免误报

### 3.3 维护期开维护模式

维护模式是设备级的报警屏蔽开关。开启后系统跳过该设备的检测，不产生新报警、不发通知、不触发 Webhook。

**何时应开启维护模式：**
- 计划停机：设备需要断电或重启
- 设备调试：更换仪表、传感器或采集器
- 网络改造：该区域网络调整，预期设备会批量离线
- 长期下线：设备临时退役

**如何开启：**
路径：系统健康 → 阈值规则 → 找到目标设备 → 编辑 → 勾选"维护模式" → 填写维护备注 → 保存

**何时关闭：** 维护结束后立即关闭。维护模式不会自动过期，忘了关会让该设备的真实故障被静默（不产生任何报警）。

**维护备注填写建议（格式）：**
```
YYYY-MM-DD HH:mm–HH:mm + 操作人 + 原因
示例："2026-04-29 22:00–24:00 张三 定期保养维护"
```

**注意事项：**
- 维护模式开启期间，已有的 ACTIVE / ACKED 报警不会被自动恢复，保持原状
- 维护期间错过的报警不会补发。关闭维护模式后，系统只对当前时刻重新检测
- 关闭维护模式时如果设备仍故障，下一轮检测会正常触发新报警（最长 60 秒延迟）

### 3.4 配置 Webhook

Webhook 把报警事件以 HTTP POST 推送给外部系统（钉钉机器人、工单系统、SIEM 等）。Webhook 与站内通知同时生效，互不影响。

**路径：** 系统健康 → Webhook 配置（仅 ADMIN 可见）

**5 个配置字段：**

| 字段 | 是否必填 | 说明 |
|------|:---:|------|
| 启用 | — | 关闭后完全跳过 Webhook 派发，站内通知不受影响 |
| URL | ✅ | 接收方地址，必须以 `http://` 或 `https://` 开头 |
| Secret | ✅ | HMAC-SHA256 签名密钥，建议使用 32 位以上随机字符串 |
| 适配器 | — | 首版默认 `generic-json`，输出原始 JSON 格式 |
| 超时（毫秒） | — | 每次 HTTP 请求的超时时长，默认 5000 毫秒 |

**"测试发送"按钮：** 点击后系统会用当前保存的配置构造一条假报警（`alarm.test` 事件）发出去，接收方返回 2xx 状态码视为成功。用来验证 URL 和 Secret 是否正确。

**下发流水（Webhook 日志）：** 页面底部显示最近一段时间的所有 Webhook 外发记录，包括每次请求的状态（SUCCESS / FAILED）、HTTP 状态码和响应时间。状态为 FAILED 的记录可点"重试"按钮手动补发。

HMAC 签名、重试策略和 payload 格式见 [alarm-webhook-integration.md](./alarm-webhook-integration.md)。

### 3.5 看健康总览

**路径：** 系统健康 → 健康总览（或点击仪表盘左上角"采集健康"卡直达）

**4 个指标卡：**

| 指标卡 | 含义 | 正常参考值 |
|--------|------|---------|
| 在线设备 | 当前有正常数据上报的设备数 | 接近总设备数 |
| 离线设备 | 超过静默超时阈值、无数据上报的设备数 | 趋近 0 |
| 报警中 | 当前处于 ACTIVE + ACKED 状态的报警数 | 趋近 0 |
| 维护中 | 当前开启了维护模式的设备数 | 按实际计划维护数量 |

**Top 10 多发设备表：** 按当前活跃报警数（ACTIVE + ACKED）倒序排列，用来发现"反复出问题"的设备。如果某台设备长期排在榜首，需要重点排查（网络质量、设备硬件老化、采集配置错误等）。

**何时该警觉 — 建议动作：**

| 信号 | 建议动作 |
|------|---------|
| 在线率 < 95% | 排查大面积离线原因；检查网络、collector 服务状态 |
| 报警中数量 > 设备总数的 5% | 报警可能已扩散为系统性问题；联系网络或运维团队 |
| Top10 中某设备 ≥ 5 条活跃报警 | 该设备可能存在持续性故障；优先派人现场检查 |
| 维护中设备数异常偏多 | 核查是否有遗忘关闭维护模式的设备 |

## 4. 常见问题（FAQ）

### Q1：为什么我没收到报警？

按以下顺序排查：

1. 角色权限：只有 ADMIN 和 OPERATOR 才会收到站内通知。VIEWER / FINANCE 不接收报警。
2. 维护模式：检查目标设备是否开启了维护模式（系统健康 → 阈值规则 → 查看该设备）。维护模式开启时跳过检测。
3. 设备是否纳入监控：设备必须在 collector 采集配置中，系统才会对其检测。不在配置里的设备不参与报警。
4. 阈值是否过宽：如果静默超时设得很大（如 7200 秒），设备要断线超过 2 小时才会报警。确认阈值是否符合预期。
5. 站内通知不推手机：站内通知只在 EMS 浏览器页面内显示（铃铛）。要手机推送的话，配 Webhook 在接收侧接入企业微信或钉钉。

---

### Q2：报警自动恢复了，我之前"确认"是不是白做了？

不是。确认（ACKED）和恢复（RESOLVED）是两个独立概念：

- ACK（确认）表示有人认领了这个报警、在跟进。这个操作会进审计日志，留下"谁、何时确认"的记录，方便复盘。
- RESOLVED（恢复）表示设备数据真的恢复了，由系统根据采集数据自动判断。

两者在报警历史中独立记录，互不影响。即使数据先于人工响应恢复，ACK 记录仍保留，审计价值不变。

---

### Q3：维护模式期间错过的报警还会补发吗？

不会。维护模式开启期间系统不做检测，不产生报警记录，关闭后也不会回填。

关闭维护模式后，系统从下一轮调度周期（最长 60 秒）开始恢复正常检测。设备此时已恢复正常的话不会有报警；仍故障则会触发新报警。

如果想知道维护期间设备的真实状态，需要人工核查（看现场记录或原始采集日志）。

---

### Q4：可以批量确认报警吗？

首版不支持。请逐条点"确认"。批量操作已进路线图（计划 v1.7+）。

在此之前，用报警历史页的筛选器（按设备或时间范围）定位要处理的报警，逐条确认。

---

### Q5：同一设备同时有多个不同类型报警怎么办？

系统对每对（设备 × 报警类型）独立维护报警状态。例如同一台设备的"静默超时"和"连续失败"是两条独立的报警记录，要分别确认和分别处理。

在报警历史页用"设备"和"类型"组合筛选，找出同一设备的所有活跃报警，逐条处理。

---

### Q6：我离职了，账号删除后我处理过的报警还能查到吗？

能查到。`alarms` 表保留 `acked_by`（确认人用户 ID），不依赖账号是否存活。历史报警列表会显示用户 ID 编号。要还原显示名，让管理员去"用户管理"按 ID 查。

审计日志（`audit_log` 表）独立保留所有写操作记录，不随账号删除而消失。

---

### Q7：改了阈值多久生效？

实时生效，不用重启服务。保存覆盖设置后，下一轮检测（最长延迟一个调度周期，默认 60 秒，由 `alarm.detection.scheduler.cron` 参数控制）就用新阈值。

要确认阈值是否生效，在阈值规则页刷新看当前生效值，也可以观察下一轮报警是否按新阈值触发。

---

### Q8：Webhook 接收方是钉钉 / 企业微信 / 飞书，要怎么接？

首版只内置 `generic-json` 适配器，向接收方发送标准 HTTP POST + JSON Body，并在请求头附带 `X-EMS-Signature` 签名字段。

钉钉、企业微信、飞书的 Webhook 接口各有专属 JSON 格式要求，与 EMS 的 generic-json 格式不同。对接方式：在接收侧搭一个轻量中转服务，收到 EMS 的 generic-json 后转换为目标平台所需格式再转发。

payload 字段说明和"接入方接收示例"代码见 [alarm-webhook-integration.md](./alarm-webhook-integration.md)。

钉钉、企业微信等平台的原生适配器已进路线图（计划 v1.7+）。

## 6. 术语词汇表

| 术语 | 解释 |
|------|------|
| **报警（Alarm）** | 系统判定某设备数据中断时产生的事件记录，包含触发时间、设备信息、处理过程等。 |
| **静默超时（Silent Timeout）** | 设备超过设定时间（默认 10 分钟）没上报任何新数据，系统判定为"数据中断"并触发报警。 |
| **连续失败（Consecutive Fail）** | 采集器对某设备连续采集失败到设定次数（默认 3 次），系统判定采集异常并触发报警。 |
| **抑制窗口（Suppression Window）** | 报警恢复后的"冷静期"（默认 5 分钟）。冷静期内即使设备再次中断也不会产生新报警，避免频繁抖动造成噪音。 |
| **维护模式（Maintenance Mode）** | 设备级的报警屏蔽开关。开启后系统跳过该设备的检测，不产生报警、不发通知。 |
| **站内通知（In-App Notification）** | 浏览器内右上角铃铛图标接收的通知，只在 EMS 页面打开时可见，不推送到手机或邮件。 |
| **Webhook** | 报警触发时向外部系统发送的 HTTP 回调请求，用于对接钉钉、工单系统等外部工具。 |
| **HMAC 签名** | 系统用 Secret 密钥对 Webhook payload 做数字签名，接收方用同一密钥验证，确认数据来源真实、未被篡改。 |
| **ACTIVE / ACKED / RESOLVED** | 报警的三个状态：ACTIVE = 活跃未处理（红色）；ACKED = 有人确认跟进（黄色）；RESOLVED = 已恢复（绿色，终态）。 |
| **AUTO / MANUAL** | 报警恢复的两种方式：AUTO = 系统检测到数据恢复后自动关闭；MANUAL = 管理员点"手动恢复"关闭。 |
