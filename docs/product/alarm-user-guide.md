# 采集中断告警 · 用户使用手册

**最近更新：** 2026-04-29
**适用版本：** v1.6.0+

本手册面向工厂运营人员（管理员和操作员），说明如何使用采集中断告警功能，包括查看告警、处理告警、配置规则以及常见问题解决。

---

## 1. 谁能做什么 — 角色对照表

本系统按角色限制操作，下表说明各角色可做什么。如发现自己缺少某个操作权限，请联系管理员。

| 操作 | 管理员 (ADMIN) | 操作员 (OPERATOR) |
|------|:---:|:---:|
| 查看告警列表与详情 | ✅ | ✅ |
| 确认告警（ACTIVE → ACKED） | ✅ | ❌ |
| 手动恢复告警（→ RESOLVED） | ✅ | ❌ |
| 配置阈值 / 维护模式 | ✅ | ❌ |
| 配置 Webhook | ✅ | ❌ |
| 接收站内通知（铃铛） | ✅ | ✅ |

> **说明**：VIEWER 和 FINANCE 角色对告警模块无访问权限，铃铛图标对其隐藏。如需查看告警，请联系管理员升级角色。

## 2. 操作员视角

### 2.1 看到铃铛上有红点该怎么办

顶部导航栏右上角的 🔔 铃铛图标带有红色角标，显示当前**活跃告警数量**（仅统计 ACTIVE 状态，每 30 秒自动刷新）。看到红点不要慌，按以下步骤逐步处理：

**处理流程（5 步）：**

1. **查看角标数量**：铃铛角标数字即为当前 ACTIVE 告警总数。角标数字为 0 时说明无待处理告警。

   > 截图占位：顶栏右上角铃铛图标，带红色角标数字"3"

2. **点击铃铛打开通知抽屉**：展开最近 20 条 ACTIVE 告警列表，按触发时间降序排列，每条显示设备名称、告警类型和触发时间。

   > 截图占位：通知抽屉展开，列出 3 条 ACTIVE 告警，每条带红色"告警"标签

3. **研判优先级**：根据"设备名称 / 告警类型 / 触发时间"判断紧急程度。触发时间越早、设备越关键，优先级越高。告警类型说明：
   - **静默超时**：设备超过设定时间没有上报新数据
   - **连续失败**：采集器对该设备连续采集失败多次

4. **确认或上报**：
   - 若您拥有 **ADMIN** 权限：可直接在抽屉中点击"确认"按钮，告警状态变为 ACKED（黄色），通知其他同事"已有人认领"。
   - 若您是 **OPERATOR**：点击"查看详情"跳转至告警历史页，将告警 ID 和设备信息告知管理员处理。

   > 截图占位：抽屉中单条告警的"确认"按钮（ADMIN 可见）与"查看详情"链接

5. **等待自动恢复**：设备数据恢复正常后，系统会在 5 分钟内自动将告警置为 RESOLVED（绿色），该条告警随即从铃铛抽屉消失，无需手动操作。

   > 截图占位：同一告警已变为绿色"已恢复"状态，从抽屉中消失

---

### 2.2 看告警历史

**路径：** 左侧菜单 → 系统健康 → 告警历史

告警历史页展示所有状态的告警记录，支持筛选和搜索。

**顶部筛选器：**

| 筛选项 | 可选值 |
|--------|--------|
| 状态 | ACTIVE（活跃）/ ACKED（已确认）/ RESOLVED（已恢复）/ 全部 |
| 类型 | 静默超时 / 连续失败 / 全部 |
| 时间范围 | 今日 / 近 7 天 / 近 30 天 / 自定义 |

**表格列含义：**

| 列名 | 含义 |
|------|------|
| 设备 | 触发告警的设备名称和 ID |
| 类型 | SILENT_TIMEOUT（静默超时）或 CONSECUTIVE_FAIL（连续失败） |
| 严重度 | 首版固定为 WARNING |
| 状态 | ACTIVE（红） / ACKED（黄） / RESOLVED（绿） |
| 触发时间 | 告警首次创建的时间 |
| 持续时长 | 从触发到当前或恢复的经过时间 |
| 操作 | 查看详情 / 确认（ADMIN）/ 手动恢复（ADMIN） |

**查看告警详情：** 点击任意行的"详情"按钮，弹出详情 Modal，内容包括：
- 完整时间线（触发 → 确认 → 恢复，每步操作人和时间）
- `detail` 原始 JSON 字段（含触发时的设备状态快照）
- 恢复方式（AUTO 自动 / MANUAL 手动）

**URL 直达：** 在浏览器地址栏输入 `/alarms/history?id=123` 可直接打开 ID 为 123 的告警详情，方便通过 IM 分享给同事。

### 2.3 看设备实时状态

有两个入口可查看单台设备的当前告警状态：

**入口一：测点管理列表**

路径：左侧菜单 → 测点管理 → 设备列表

列表中的"告警"列显示三种状态：
- **未启用**：该设备未纳入告警监控（不在 collector 配置中）
- **告警中**（红色徽章）：当前存在 ACTIVE 或 ACKED 告警
- **正常**（绿色徽章）：无活跃告警

**入口二：仪表盘**

路径：左侧菜单 → 仪表盘

- **左上角"采集健康"卡**：显示在线率百分比 + 当前告警总数。点击该卡跳转至健康总览页。
- **Top N 图表**：点击具体设备名称，展开设备抽屉，顶部显示状态徽章，下方"最近告警"Timeline 列出该设备最近 5 条告警（颜色编码：红 = ACTIVE，黄 = ACKED，绿 = RESOLVED）。

## 3. 管理员视角

### 3.1 处理一条新告警

从告警触发到关闭的完整操作流程：

> 截图 1：铃铛角标从 0 变为 +1，红色角标出现

**第一步：发现告警**
铃铛角标出现或数字增加，说明有新的 ACTIVE 告警产生。系统同时推送站内通知，并（如已配置）触发 Webhook 向外部系统发送通知。

> 截图 2：通知抽屉打开，列出新的 ACTIVE 告警，红色"告警"标签，显示设备名和触发时间

**第二步：打开抽屉确认情况**
点击铃铛打开通知抽屉，查看是哪台设备、什么类型的告警、何时触发。判断是真实故障还是短暂抖动（通常 5 分钟后会自动恢复）。

**第三步：点击"确认"认领告警**
确认后告警状态从 ACTIVE（红）变为 ACKED（黄），铃铛角标数字随之减少。这向所有人发出信号：此告警已有人负责跟进，避免重复响应。

> 截图 3：点击"确认"按钮前后对比，状态从红色 ACTIVE 变为黄色 ACKED

> **重要**：**确认 ≠ 已解决。** 确认（ACKED）仅表示有人在跟进，问题尚未修复。系统会记录确认时间和确认人，方便审计。

**第四步：等待设备恢复**
派遣维修人员或联系网络团队处理。设备数据恢复正常后，系统在 5 分钟内自动将告警置为 RESOLVED。

> 截图 4：告警状态变为绿色"已恢复"，`resolved_reason = AUTO`，通知抽屉中该条消失

**第五步（可选）：手动关闭**
若现场已确认设备修复完毕但数据尚未上报，或业务决定不再等待，ADMIN 可直接点击"手动恢复"立即关闭告警，记录 `resolved_reason = MANUAL`。

**三态转换说明：**

```
ACTIVE（红） ──[ADMIN 点确认]──────────────────────────────→ ACKED（黄）
                                                              │
                                         [数据恢复 5min / 手动恢复]
                                                              ↓
ACTIVE（红） ──[数据恢复 5min]────────────────────────────→ RESOLVED（绿）
ACTIVE（红） ──[ADMIN 点手动恢复]─────────────────────────→ RESOLVED（绿）
```

RESOLVED 是终态，不可再转换。若设备再次中断，系统将创建一条全新的告警记录。

### 3.2 给某台设备单独配置阈值

**路径：** 系统健康 → 阈值规则 → 设备列表 → 找到目标设备 → 点击"编辑"

**全局默认值（只读）：**

页面顶部显示当前全局配置，这些值由后端 `alarm.detection.*` 参数控制，在此页面只能查看，不能修改。如需调整全局值，请联系系统管理员修改服务器配置。详情参见 [alarm-config-reference.md](./alarm-config-reference.md)。

| 全局参数 | 默认值 | 含义 |
|---------|--------|------|
| 静默超时秒数 | 600 秒 | 超过 10 分钟无数据则触发 |
| 连续失败次数 | 3 次 | 连续采集失败 3 次则触发 |
| 抑制窗口秒数 | 300 秒 | 告警恢复后 5 分钟内不重复触发 |

**设备级覆盖（可编辑）：**

在设备列表中找到目标设备，点击"编辑"，可单独设置以下字段：

| 字段 | 说明 | 留空效果 |
|------|------|---------|
| 静默超时（秒） | 该设备专属的超时时长 | 留空 = 沿用全局默认 |
| 连续失败次数 | 该设备专属的失败阈值 | 留空 = 沿用全局默认 |
| 维护模式 | 开关，开启时跳过告警检测 | 默认关闭 |
| 维护备注 | 说明维护原因和计划时间 | 可选，建议填写 |

**生效时间：** 保存后**立即生效**，最长延迟一个调度周期（默认 60 秒）。下一轮检测即使用新阈值。

**常用场景：**
- 高频采集设备（每 5 秒一次）：将静默超时调低至 60 秒，更快感知中断
- 网络不稳定设备：将连续失败次数调高至 10 次，减少因抖动产生的误报
- 低频采集设备（每小时一次）：将静默超时调高至 3600 秒，避免误报

### 3.3 维护期开维护模式

维护模式是设备级的告警屏蔽开关。开启后，系统完全跳过该设备的告警检测，不产生新告警、不发通知、不触发 Webhook。

**何时应开启维护模式：**
- 计划停机：设备需要断电或重启
- 设备调试：更换仪表、传感器或采集器
- 网络改造：该区域网络调整，预期设备会批量离线
- 长期下线：设备临时退役

**如何开启：**
路径：系统健康 → 阈值规则 → 找到目标设备 → 编辑 → 勾选"维护模式" → 填写维护备注 → 保存

**何时关闭：** 维护结束后**立即关闭**。维护模式不会自动过期，忘记关闭会导致该设备的真实故障被静默（不产生任何告警）。

**维护备注填写建议（格式）：**
```
YYYY-MM-DD HH:mm–HH:mm + 操作人 + 原因
示例："2026-04-29 22:00–24:00 张三 定期保养维护"
```

**重要注意事项：**
- 维护模式开启期间，已有的 ACTIVE / ACKED 告警**不会被自动恢复**，保持原状
- 维护期间**错过的告警不会补发**。关闭维护模式后，系统仅对当前时刻重新检测
- 若关闭维护模式时设备仍处于故障状态，下一轮检测会正常触发新告警（最长 60 秒延迟）

### 3.4 配置 Webhook

Webhook 将告警事件以 HTTP POST 方式推送给外部系统（钉钉机器人、工单系统、SIEM 等）。Webhook 与站内通知同时生效，互不影响。

**路径：** 系统健康 → Webhook 配置（仅 ADMIN 可见）

**5 个配置字段：**

| 字段 | 是否必填 | 说明 |
|------|:---:|------|
| 启用 | — | 关闭后完全跳过 Webhook 派发，站内通知不受影响 |
| URL | ✅ | 接收方地址，必须以 `http://` 或 `https://` 开头 |
| Secret | ✅ | HMAC-SHA256 签名密钥，建议使用 32 位以上随机字符串 |
| 适配器 | — | 首版默认 `generic-json`，输出原始 JSON 格式 |
| 超时（毫秒） | — | 每次 HTTP 请求的超时时长，默认 5000 毫秒 |

**"测试发送"按钮：** 点击后，系统立即使用当前保存的配置构造一条假告警（`alarm.test` 事件）发出，接收方返回 2xx 状态码视为成功。可用于验证 URL 和 Secret 是否正确。

**下发流水（Webhook 日志）：** 页面底部显示最近一段时间的所有 Webhook 外发记录，包含每次请求的状态（SUCCESS / FAILED）、HTTP 状态码和响应时间。状态为 FAILED 的记录可点击"重试"按钮手动触发补发。

**关于 HMAC 签名、重试策略和 payload 格式**，请参阅 [alarm-webhook-integration.md](./alarm-webhook-integration.md)。

### 3.5 看健康总览

**路径：** 系统健康 → 健康总览（或点击仪表盘左上角"采集健康"卡直达）

**4 个指标卡：**

| 指标卡 | 含义 | 正常参考值 |
|--------|------|---------|
| 在线设备 | 当前有正常数据上报的设备数 | 接近总设备数 |
| 离线设备 | 超过静默超时阈值、无数据上报的设备数 | 趋近 0 |
| 告警中 | 当前处于 ACTIVE + ACKED 状态的告警数 | 趋近 0 |
| 维护中 | 当前开启了维护模式的设备数 | 按实际计划维护数量 |

**Top 10 多发设备表：** 按当前活跃告警数（ACTIVE + ACKED）倒序排列，用于发现"反复出问题"的设备。如果某台设备长期排在榜首，说明需要重点排查（网络质量、设备硬件老化、采集配置错误等）。

**何时该警觉 — 建议动作：**

| 信号 | 建议动作 |
|------|---------|
| 在线率 < 95% | 排查大面积离线原因；检查网络、collector 服务状态 |
| 告警中数量 > 设备总数的 5% | 告警可能已扩散为系统性问题；联系网络或运维团队 |
| Top10 中某设备 ≥ 5 条活跃告警 | 该设备可能存在持续性故障；优先派人现场检查 |
| 维护中设备数异常偏多 | 核查是否有遗忘关闭维护模式的设备 |

## 4. 常见问题（FAQ）

### Q1：为什么我没收到告警？

按以下顺序排查：

1. **角色权限**：只有 ADMIN 和 OPERATOR 角色才能收到站内通知。VIEWER / FINANCE 不接收告警。
2. **维护模式**：检查目标设备是否开启了维护模式（系统健康 → 阈值规则 → 查看该设备）。维护模式开启时完全跳过检测。
3. **设备是否纳入监控**：设备必须在 collector 采集配置中，系统才会对其检测。未在配置中的设备不参与告警。
4. **阈值是否过宽**：如果静默超时设置得很大（如 7200 秒），只有设备断线超过 2 小时才会告警。确认阈值是否符合预期。
5. **站内通知不推手机**：站内通知仅在 EMS 浏览器页面内显示（铃铛）。如需手机推送，请配置 Webhook 并在接收侧接入企业微信/钉钉等。

---

### Q2：告警自动恢复了，我之前"确认"是不是白做了？

不是。确认（ACKED）和恢复（RESOLVED）是两个独立的概念：

- **ACK（确认）** 表示有人认领了这个告警、正在跟进处理。这个操作会被记入审计日志，留下"谁、何时确认"的记录，方便复盘。
- **RESOLVED（恢复）** 表示设备数据已真正恢复，是系统根据采集数据自动判断的。

两者都独立记录在告警历史中，互不影响。即使数据先于人工响应恢复，ACK 记录依然保留，审计价值不变。

---

### Q3：维护模式期间错过的告警还会补发吗？

**不会。** 维护模式开启期间系统完全不做检测，不产生任何告警记录，也不会在关闭后回填。

关闭维护模式后，系统从下一轮调度周期（最长 60 秒）开始恢复正常检测。如果设备此时已恢复正常，不会有任何告警。如果设备仍处于故障状态，会触发新告警。

如果需要了解维护期间设备的真实状态，需要通过人工核查（如查看现场记录或原始采集日志）。

---

### Q4：可以批量确认告警吗？

**首版不支持。** 请逐条点击"确认"。批量操作功能已纳入路线图（计划 v1.7+）。

在此之前，可通过告警历史页的筛选器（按设备 / 时间范围）快速定位需要处理的告警，逐条确认。

---

### Q5：同一设备同时有多个不同类型告警怎么办？

系统对每对**（设备 × 告警类型）**独立维护告警状态。例如，同一台设备的"静默超时"和"连续失败"是两条独立的告警记录，需要**分别确认**和**分别处理**。

在告警历史页可用"设备"和"类型"组合筛选，找出同一设备的所有活跃告警，逐条处理。

---

### Q6：我离职了，账号删除后我处理过的告警还能查到吗？

**能查到。** `alarms` 表保留 `acked_by`（确认人用户 ID），不依赖账号是否存活。历史告警列表会显示用户 ID 编号。如需还原显示名，请管理员前往"用户管理"按 ID 查询。

审计日志（`audit_log` 表）独立保留所有写操作记录，不随账号删除而消失。

---

### Q7：改了阈值多久生效？

**实时生效**，无需重启服务。保存覆盖设置后，下一轮检测（最长延迟一个调度周期，默认 60 秒，由 `alarm.detection.scheduler.cron` 参数控制）即使用新阈值。

如需确认阈值是否已生效，可在阈值规则页刷新查看当前生效值，也可观察下一轮告警是否按新阈值触发。

---

### Q8：Webhook 接收方是钉钉 / 企业微信 / 飞书，要怎么接？

首版只内置 `generic-json` 适配器，向接收方发送标准 HTTP POST + JSON Body，并在请求头附带 `X-EMS-Signature` 签名字段。

钉钉、企业微信、飞书的 Webhook 接口各有专属 JSON 格式要求，与 EMS 的 generic-json 格式不同。对接方式：在接收侧搭建一个轻量中转服务，接收 EMS 的 generic-json，转换为目标平台所需格式后再转发。

具体 payload 字段说明和"接入方接收示例"代码，请参阅 [alarm-webhook-integration.md](./alarm-webhook-integration.md)。

钉钉、企业微信等平台的原生适配器已纳入路线图（计划 v1.7+）。

## 6. 术语词汇表

| 术语 | 解释 |
|------|------|
| **告警（Alarm）** | 系统判定某设备数据中断时自动产生的事件记录，包含触发时间、设备信息、处理过程等。 |
| **静默超时（Silent Timeout）** | 设备超过设定时间（默认 10 分钟）没有上报任何新数据，系统判定为"数据中断"并触发告警。 |
| **连续失败（Consecutive Fail）** | 采集器对某设备连续采集失败达到设定次数（默认 3 次），系统判定采集异常并触发告警。 |
| **抑制窗口（Suppression Window）** | 告警恢复后的"冷静期"（默认 5 分钟）。冷静期内即使设备再次中断，也不会产生新告警，避免频繁抖动造成告警噪音。 |
| **维护模式（Maintenance Mode）** | 设备级的告警屏蔽开关。开启后系统完全跳过该设备的检测，不产生告警、不发通知。 |
| **站内通知（In-App Notification）** | 浏览器内右上角铃铛图标接收的通知，仅在 EMS 页面打开时可见，不推送至手机或邮件。 |
| **Webhook** | 告警触发时向外部系统发送的 HTTP 回调请求，可用于对接钉钉、工单系统等外部工具。 |
| **HMAC 签名** | 系统用 Secret 密钥对 Webhook payload 进行数字签名，接收方用同一密钥验证，确认数据来源真实可信、未被篡改。 |
| **ACTIVE / ACKED / RESOLVED** | 告警的三个状态：ACTIVE = 活跃未处理（红色）；ACKED = 已有人确认跟进（黄色）；RESOLVED = 已恢复（绿色，终态）。 |
| **AUTO / MANUAL** | 告警恢复的两种方式：AUTO = 系统检测到数据恢复后自动关闭；MANUAL = 管理员主动点击"手动恢复"关闭。 |
