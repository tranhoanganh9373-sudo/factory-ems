## 1. 状态机概览

每条采集中断报警有三种状态：ACTIVE（活跃）、ACKED（已确认）、RESOLVED（已恢复）。状态由两类事件驱动：系统自动（检测命中、数据恢复）和用户手动（确认、手动恢复）。

```
                       ┌────────────────┐
                       │   （无报警）   │
                       └────────┬───────┘
                                │ 检测命中（且无活动报警）
                                ▼
                      ┌──────────────────┐
                      │     ACTIVE       │
                      │  红色 Tag        │
                      │  铃铛角标 +1     │
                      │  Webhook：触发   │
                      │  站内：TRIGGERED │
                      └─┬────────────┬───┘
                        │            │
    用户点击"确认"      │            │ 数据恢复 + 距 triggered > 5min
    （仅 ADMIN）        │            │
                        ▼            ▼
              ┌──────────────┐  ┌──────────────────┐
              │   ACKED      │  │   RESOLVED       │
              │  黄色 Tag    │  │  绿色 Tag        │
              │  铃铛不计数  │  │  resolved_reason │
              │  Webhook：无 │  │   = AUTO         │
              │  站内：无    │  │  站内：RESOLVED  │
              └──────┬───────┘  └──────────────────┘
                     │
                     │ ① 数据恢复 + 距 triggered > 5min → RESOLVED (AUTO)
                     │ ② 用户点击"手动恢复"（仅 ADMIN）→ RESOLVED (MANUAL)
                     ▼
                  RESOLVED
                  绿色 Tag
              ┌───────────────┐
              │ resolved_reason│
              │  = MANUAL      │
              └───────────────┘
              ↑
              用户从 ACTIVE 直接点"手动恢复"（ADMIN）→ RESOLVED (MANUAL)
```

AUTO 与 MANUAL 两种 RESOLVED 路径的区别：

- AUTO：系统检测到设备数据恢复正常，且距报警触发已超过 5 分钟抑制窗口，后端自动将报警置为 RESOLVED，`resolved_reason = AUTO`，同时推送站内通知 `RESOLVED`。不需要人工介入。
- MANUAL：ADMIN 用户点击"手动恢复"按钮，无论当前是 ACTIVE 还是 ACKED 都可以转为 RESOLVED，`resolved_reason = MANUAL`。常见情形：运维已确认设备修复但数据还没上报，或业务方决定不再追踪该条报警。

## 2. 三种状态详解

### ACTIVE

| 维度 | 说明 |
|------|------|
| **UI 表现** | 红色 Tag「报警」；全局铃铛角标 +1；出现在通知中心抽屉最新 20 条 |
| **触发条件** | 检测命中（静默超时 / 连续失败），且该设备该类型当前无 ACTIVE/ACKED 报警 |
| **用户可操作** | ADMIN：点击"确认"→ ACKED；点击"手动恢复"→ RESOLVED (MANUAL)；OPERATOR：仅查看 |
| **后端动作** | 写入 alarms 表；触发 Webhook（带 TRIGGERED 事件）；推送站内通知 TRIGGERED |
| **通常持续时间** | 数秒到数分钟，取决于运维响应速度；无人确认时维持直到自动恢复 |

### ACKED

| 维度 | 说明 |
|------|------|
| **UI 表现** | 黄色 Tag「已确认」；铃铛角标不再计入此条；仍显示在健康总览"报警中"卡 |
| **触发条件** | ADMIN 对 ACTIVE 报警点击"确认"操作 |
| **用户可操作** | ADMIN：点击"手动恢复"→ RESOLVED (MANUAL)；OPERATOR：仅查看 |
| **后端动作** | 更新 alarms 表 status = ACKED，记录 acked_at / acked_by；不发 Webhook；不推站内通知 |
| **通常持续时间** | 视维修进度而定，从几分钟到几小时不等；数据恢复后 5min 自动转 RESOLVED |

### RESOLVED

| 维度 | 说明 |
|------|------|
| **UI 表现** | 绿色 Tag「已恢复」；不在铃铛角标计数；仅在报警历史页或设备报警时间线可见 |
| **触发条件** | ① AUTO：数据恢复且距 triggered_at > 5min；② MANUAL：ADMIN 手动操作 |
| **用户可操作** | 仅查看（终态，不可再转换） |
| **后端动作** | 更新 status = RESOLVED，记录 resolved_at / resolved_by / resolved_reason；推站内通知 RESOLVED（AUTO 路径）；进入抑制窗口（5min 内不再触发同类型报警） |
| **通常持续时间** | 永久终态；5min 抑制窗口结束后允许同类型重新触发 |

### 各页面对状态展示

| 页面 | 显示哪些状态 | 排序 |
|------|-------------|------|
| 全局铃铛角标 | count(ACTIVE) | — |
| 通知中心抽屉 | 最新 20 条 ACTIVE | triggered_at DESC |
| 健康总览→"报警中"卡 | count(ACTIVE) + count(ACKED) | — |
| 报警历史页 | 全部状态（支持筛选切换） | triggered_at DESC |
| 设备详情→报警时间线 | 该设备最近 5 条全状态 | triggered_at DESC |

## 3. 阈值解析规则

每台设备的阈值有两层：设备级覆盖（alarm_rules_override 表）优先于全局默认（Spring Boot 配置文件中的 alarm-rules 参数）。回落是逐字段独立的：某字段在设备覆盖中为 NULL 或无记录，只有该字段用全局默认，其他已覆盖字段不受影响。

### 字段覆盖规则表

| 阈值字段 | 全局默认参数 | 设备覆盖字段 | 回落规则 |
|---------|------------|------------|---------|
| 静默超时（秒） | `default-silent-timeout-seconds`（如 600） | `silent_timeout_seconds` | 覆盖记录该字段非 NULL → 用覆盖值；NULL 或无记录 → 用全局 |
| 连续失败次数 | `default-consecutive-fail-count`（如 3） | `consecutive_fail_count` | 覆盖记录该字段非 NULL → 用覆盖值；NULL 或无记录 → 用全局 |
| 维护模式 | 固定为 `false` | `maintenance_mode` | 覆盖记录字段值；无记录 → false |

ThresholdResolver 返回的最终生效值以 record `Resolved(silentTimeoutSeconds, consecutiveFailCount, maintenanceMode)` 形式供检测逻辑使用。

### 部分覆盖示例

**示例 1：仅覆盖静默超时**

设备 A 在 alarm_rules_override 中设置了 `silent_timeout_seconds = 120`，`consecutive_fail_count` 字段为 NULL。

| 字段 | 最终生效值 | 来源 |
|------|----------|------|
| silentTimeoutSeconds | **120 秒** | 设备覆盖 |
| consecutiveFailCount | **3 次**（全局默认） | 全局回落 |
| maintenanceMode | **false** | 全局回落 |

**示例 2：仅开启维护模式**

设备 B 设置了 `maintenance_mode = true`，`silent_timeout_seconds` 和 `consecutive_fail_count` 均为 NULL。

| 字段 | 最终生效值 | 来源 |
|------|----------|------|
| silentTimeoutSeconds | **600 秒**（全局默认） | 全局回落（维护期间阈值实际不再使用） |
| consecutiveFailCount | **3 次**（全局默认） | 全局回落 |
| maintenanceMode | **true** | 设备覆盖 |

**示例 3：完全无覆盖记录**

设备 C 在 alarm_rules_override 表中不存在任何记录。

| 字段 | 最终生效值 | 来源 |
|------|----------|------|
| silentTimeoutSeconds | **600 秒**（全局默认） | 全局回落 |
| consecutiveFailCount | **3 次**（全局默认） | 全局回落 |
| maintenanceMode | **false** | 全局回落 |

### 验收 SQL

查询设备 88 的覆盖记录，判断每字段最终生效值：

```sql
SELECT * FROM alarm_rules_override WHERE device_id = 88;
```

读取结果时：
- 若无行返回 → 三个字段全部使用全局默认；
- 若有行返回 → `silent_timeout_seconds` 非 NULL 时用覆盖值，否则用全局；`consecutive_fail_count` 同理；`maintenance_mode` 直接读字段值。

## 4. 抑制窗口（防抖）

抑制窗口用于防止报警"抖动"。默认 5 分钟（300 秒），由全局参数 `suppression-window-seconds` 控制，可调整。

### (a) RESOLVED 后 5 分钟不再触发新报警

报警转为 RESOLVED 后，同一设备同一类型在 5 分钟内不会再次触发，避免"数据刚恢复又断"在列表里反复刷屏。

```
时间线：

T0        T1        T2=T1+5min   T3
|----ACTIVE----|----RESOLVED----|============抑制窗口============|---可再次触发
              数据恢复          RESOLVED
                                写入

↑ 若 T2 之前又中断：         不会创建新 ACTIVE（抑制窗口生效）
↑ 若 T3 之后又中断：         正常创建新 ACTIVE
```

### (b) ACTIVE 触发后 5 分钟内不允许 AUTO 自动恢复

报警触发（ACTIVE）后，即使数据立即恢复正常，系统也不会在 5 分钟内自动将其置为 RESOLVED，避免"瞬时恢复 → RESOLVED → 立刻又 ACTIVE"的循环抖动。

```
时间线：

T0          T0+30s        T0+5min      T0+5min+检测周期
|---ACTIVE---|---数据恢复---|=========等待抑制窗口========|---AUTO RESOLVED
             ↑                         ↑
        数据已正常                  此时才允许
        但不会立刻 RESOLVED         AUTO 自动恢复
```

### 调优建议

| 设备类型 | 建议值 | 原因 |
|---------|--------|------|
| 网络抖动频繁的设备 | 10–30 分钟（600–1800 秒） | 减少抖动噪音 |
| 稳定环境下的设备 | 1–2 分钟（60–120 秒） | 更快响应真实恢复 |
| 默认值 | 5 分钟（300 秒） | 适合大多数工厂场景 |

修改示例（`application.yml`）：

```yaml
alarm-rules:
  suppression-window-seconds: 600   # 调整为 10 分钟
```

## 5. 维护模式完整流程

### 定义

维护模式是设备级的报警屏蔽开关。开启后，整个维护期间系统跳过该设备的检测和恢复逻辑，避免计划停机在报警列表里制造无效噪音。

### 何时应开启维护模式

以下场景建议在操作开始前开启维护模式：

1. 计划停机：设备因生产排程需要断电或重启；
2. 设备搬迁：设备从一个车间迁移到另一个位置，期间必然断线；
3. 定期保养维护：设备做例行保养，数据采集会中断若干小时；
4. 网络改造：网络架构调整导致该区域设备批量离线；
5. 长期下线：设备临时退役，短期内不会恢复生产；
6. 报警调试：系统工程师反复触发与清除报警做调试测试。

### 操作步骤

**UI 路径**：系统健康 → 阈值规则 → 设备列表 → 找到目标设备 → 勾选"维护模式" → 填写 maintenance_note → 保存

**API 路径**：

```http
PUT /api/v1/alarm-rules/overrides/{deviceId}
Content-Type: application/json

{
  "maintenanceMode": true,
  "maintenanceNote": "2026-04-29 09:00–11:00 定期保养，联系人：张三，预计 11:00 恢复"
}
```

**maintenance_note 填写规范**（建议格式）：

```
{原因} + {计划恢复时间} + {联系人}
示例："生产线 B 轴承更换，计划 2026-04-29 11:00 恢复，联系：张三 13800138000"
```

### 维护期间系统行为

| 行为维度 | 维护模式开启时 |
|---------|--------------|
| 检测逻辑 | 完全跳过，不触发新报警 |
| 已有 ACTIVE/ACKED 报警 | 保持原状，不会被自动恢复（维护模式不影响现存报警状态） |
| 站内通知 | 不发送 |
| Webhook | 不触发 |

### 关闭维护模式后

- 取消勾选"维护模式"并保存后，下一轮检测周期（≤ 60 秒）开始重新检测；
- 期间错过的报警不会补发：维护期间的断线历史不回填到报警记录，关闭后只对当前时刻的状态重新检测；
- 若关闭时设备仍离线，从下一轮检测起正常触发报警。

## 6. 边界场景说明

以下是生产环境中常见的异常或特殊情况，提前了解这些行为可以减少误判和不必要的操作。

| 场景 | 系统行为 | 设计原因 |
|------|---------|---------|
| 同设备同类型重复触发（已 ACTIVE/ACKED） | 不再创建新行，仅更新 detail JSONB 字段 | 避免同一故障产生多条重复报警，保持列表整洁 |
| 触发后立即恢复（< 5min） | 不会自动恢复（抑制窗口生效），等满 5min 后才 AUTO RESOLVED | 防止因瞬时抖动导致报警立即消失，运维未能及时感知 |
| RESOLVED 后立即又中断（< 5min） | 不会触发新报警（抑制窗口生效），等满 5min 后才允许新 ACTIVE | 防止网络抖动在短时间内产生大量反复报警 |
| 设备从未上报（last_seen IS NULL） | 不触发 SILENT_TIMEOUT 类型报警；连续失败类型报警仍可正常触发 | 静默超时以"上次上报时间"为基准，无记录则无法计算超时 |
| maintenance_mode = true | 完全跳过该设备（不触发 / 不自动恢复 / 现存报警保持原状） | 维护期间一切检测暂停，避免计划性停机产生噪音 |
| collector 重启 | 内存中的 consecutiveErrors 清零，CONSECUTIVE_FAIL 类型报警可能延迟 N 个周期才再次触发；SILENT_TIMEOUT 不受影响 | 连续失败计数依赖内存状态，重启后需重新积累；静默超时依赖数据库时间戳，不依赖内存 |

## 7. 用户应对场景手册

下面 7 个是生产环境中的典型场景，说明系统会做什么、用户该做什么或不必做什么。

---

### 场景 1：设备真的坏了（断电 / 硬件故障）

**场景描述**：产线上一台数据采集器因断电完全停止上报，持续超过静默超时阈值（默认 600 秒）。

**系统会做什么**：
- 检测命中，创建 ACTIVE 报警（类型 SILENT_TIMEOUT 或 CONSECUTIVE_FAIL）；
- 触发 Webhook 通知；
- 推送站内通知，铃铛角标 +1。

**用户该做什么**：
1. ADMIN 点击"确认"，报警变为 ACKED，铃铛角标停止叠加；
2. 派人检修，修复后设备恢复上报；
3. 系统检测到数据恢复后，5 分钟内自动 RESOLVED，不用手动操作；
4. 若维修耗时较长，可提前在 UI 开启维护模式，避免重复报警。

---

### 场景 2：网络抖动 30 秒

**场景描述**：工厂网络短暂波动，某台设备断线 30 秒后恢复。

**系统会做什么**：
- 若 30 秒内连续失败次数没达到阈值（默认 3 次轮询），不会触发报警；
- 若恰好达到阈值但 30 秒后已恢复，因抑制窗口（5min）生效，也不会触发新报警。

**用户该做什么**：
- 不用操作，系统会过滤短暂抖动；
- 若此类设备抖动频繁导致报警噪音，可调大该设备的 `silent_timeout_seconds` 或全局 `suppression-window-seconds`。

---

### 场景 3：计划停机维护 2 小时

**场景描述**：某设备需要做每月定期保养，计划今晚 22:00–24:00 停机。

**系统会做什么（未开维护模式）**：
- 停机后约 10 分钟内触发报警，产生 ACTIVE；
- 报警发送 Webhook 通知相关人员，可能打扰夜班运维。

**用户该做什么**：
1. 停机前：在 UI 为该设备开启维护模式，填写备注"保养维护，22:00–24:00，联系：张三"；
2. 维护期间：系统不触发任何报警；
3. 维护完成后：关闭维护模式，系统从下一轮检测周期（≤ 60 秒）恢复监控；
4. 若设备已正常上报数据，不会有报警。

---

### 场景 4：误开维护模式后忘记关

**场景描述**：运维人员开启了某设备的维护模式，但维护完成后忘记关闭，设备此后出现真实故障也没报警。

**系统会做什么**：
- 维护模式开启期间，无论设备是否真实故障，都不触发报警；
- 故障被静默，不进入报警列表。

**用户该做什么**：
1. 发现问题后立即在 UI 关闭维护模式；
2. 系统从下一个检测周期起恢复检测（≤ 60 秒生效）；
3. 若设备当前仍故障，正常触发新报警；
4. 建议：开启维护模式时填写明确的计划恢复时间，并给自己设个日历提醒。

---

### 场景 5：同一设备反复触发（5 分钟内多次）

**场景描述**：某设备网络极不稳定，短时间内反复上下线，报警列表频繁刷新。

**系统会做什么**：
- 第一次命中时创建 ACTIVE 报警；
- 后续同类型重复触发时，不创建新报警，只更新已有报警的 detail 字段；
- RESOLVED 后 5 分钟内再次中断，不触发新报警（抑制窗口生效）。

**用户该做什么**：
1. 不用手动处理重复报警，系统会自动合并；
2. 调大该设备的 `consecutive_fail_count`（如从 3 次增至 10 次）或 `suppression-window-seconds`（如增至 1800 秒）减少噪音；
3. 联系网络团队排查抖动根因。

---

### 场景 6：收到报警后已派人去修，但还需 30 分钟到达

**场景描述**：报警触发后，ADMIN 已安排人员前往现场检修，但人员还在途中，30 分钟后才能到达。

**系统会做什么**：
- 报警保持 ACTIVE，铃铛角标继续计数；
- 其他运维人员看到通知可能会重复响应。

**用户该做什么**：
1. 立即点"确认"将报警转为 ACKED，报警变黄色，铃铛角标停止叠加；
2. 这相当于在群里告诉所有人：此报警已有人认领，不用重复响应；
3. 人员抵达现场修复后，设备恢复上报，5 分钟后系统自动 RESOLVED；
4. 也可以在人员到达后点"手动恢复"立即关闭报警（MANUAL）。

---

### 场景 7：多人同时收到通知，谁来处理

**场景描述**：同一报警触发时，多名 ADMIN 同时收到站内通知和 Webhook，谁来响应没说清楚。

**系统会做什么**：
- 记录第一个点"确认"操作的用户（acked_by 字段）；
- 后续其他人看到报警已变为 ACKED（黄色），就知道有人认领了；
- 同一报警只能被确认一次，不会出现重复 ACKED。

**用户该做什么**：
1. 约定响应优先级（如按区域或班次），减少混乱；
2. 第一个看到报警的 ADMIN 立即点"确认"，明确认领；
3. 其他人看到黄色 ACKED 后，找认领人核实是否需要协助；
4. 若没人在 5 分钟内响应，可在 Webhook 中配置升级通知（通过 escalation 规则）。
