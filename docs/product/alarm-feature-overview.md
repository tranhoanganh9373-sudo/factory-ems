# 采集中断报警 · 功能概览

> 适用版本：v1.6.0+ ｜ 最近更新：2026-05-04

---

## §1 一句话价值

实时监测每台设备的数据采集状态，断了第一时间通知，避免数据缺失影响计费、报表、能效分析。
不用人工盯盘，报警到手、处理有记录、恢复有闭环。

---

## §2 解决什么问题

- **仪表掉线没感知，等数据丢了才发现**
  仪表断电、网络中断后，系统不报警、运维不知道，等月底对账时才发现数据缺失，补录成本高、精度差。

- **采集程序卡死或网络抖动，KPI 报表出现断点**
  采集进程假死或网络短暂抖动时，曲线数据出现空洞，能耗分析、高峰核查等 KPI 报表失真，影响决策。

- **维护期间报警泛滥，运维团队产生"狼来了"麻木**
  设备计划停机或例行维护时，报警持续轰炸，团队逐渐忽视通知，真正出问题时反而错过响应窗口。

- **多套上游系统要同步通知，信息孤岛打不通**
  运维团队用钉钉，IT 部门用企业微信，工单系统又是另一套，每次异常都要手动广播，容易遗漏。

- **不同设备业务重要度不同，统一阈值不够用**
  关键主变压器和普通照明回路对断线容忍度差异巨大，一刀切的超时阈值要么漏报要么误报。

## §3 核心功能

- **双口径检测**：支持"静默超时"（多久没收到数据）和"连续失败"（连续多少次采集错误）两种检测口径，满足其一就触发报警，覆盖更多断线场景。
- **状态机闭环**：报警经过"触发 → 已确认 → 已解决"完整流程，支持系统自动恢复和人工手动关闭，每次操作留审计记录。
- **设备级阈值覆盖 + 维护模式**：可对单台设备单独配置超时阈值，开启维护模式期间自动屏蔽报警，操作要填写备注，全程可审计。
- **站内通知**：管理员和运维人员登录后看得到铃铛角标和通知抽屉，系统每 30 秒轮询一次，不需要额外客户端。
- **Webhook 外发**：报警触发时推送到企业自定义地址，载荷用 HMAC-SHA256（消息签名标准）保证安全，发送失败会自动重试并记录下发流水。
- **健康总览看板**：一屏展示全场设备的采集健康状态，内置"近 7 天报警最多 Top10 设备"排行，快速定位高频异常源。
- **真实在线状态判定**（v1.1.0+）：基于 `DiagnosticRingBuffer.lastGoodSampleAt` + 5 分钟新鲜度窗口判断在线/离线，不再依赖 enabled 标志，与表计列表页「采集状态」列同口径。维护模式下走独立分支，避免被误判离线。
- **表计粒度状态端点**（v1.1.0+）：`GET /api/v1/alarms/meter-status` 返回 `meterId → ONLINE | OFFLINE | MAINTENANCE` 映射，供 `/meters` 列表页联表渲染状态列；与 `/health-summary` 卡共享同一计算逻辑，保证一屏内每个数字对得上。

## §4 适用场景

### 场景 A：能源管理项目方 — 保障报表数据完整性

能源审计或碳核查项目通常对数据完整率有硬性要求（如不低于 95%）。部署采集中断报警后，仪表断线 5 分钟内就收到通知，运维人员快速响应、补采集窗口，避免月底对账时发现大段数据缺失。对重要度高的主回路，单独把超时阈值调低，做到"漏一个点都不放过"；次要回路则适当放宽，减少噪音。

### 场景 B：工厂自动化运维团队 — 区分维护期和真实异常

大型工厂有定期停机计划，逢节假日或设备年检，几十台仪表会同时离线。运维负责人在停机前为相关设备开启"维护模式"，报警自动暂停，维护结束后关闭即可恢复监测。真正的非计划断线照常报警，团队不会因为维护期间的噪音报警降低对真实故障的敏感度。

### 场景 C：多系统集成 — Webhook 桥接，不另开通道

企业内已有钉钉机器人、企业微信群或 ITSM 工单平台的，把 Webhook 地址配成对应系统的入口即可。报警触发时 ems-alarm 带签名推送，接收方不用改造，一个 Webhook 地址就把报警同步到现有通知渠道，不需要为能源管理再单独部署报警中间件。

## §5 不在范围（首版）

以下能力在首个发布版本（v1.6.0）里还不支持，已进入后续路线图，购买前请与项目方确认：

- **邮件 / 短信 / IM 直连**：首版只提供 Webhook 推送，钉钉、企业微信等 IM 接收方要通过 Webhook 桥接自己对接，不内置直连。
- **多个 Webhook 端点**：首版只支持配置一个全局 Webhook 地址，多目标分发要在接收方做转发。
- **报警严重程度分级**：首版所有报警统一为普通级别（NORMAL），不支持"紧急 / 警告 / 提示"分级，该能力规划在 v1.7+。
- **报警 SLA 计算与升级路由**：首版不提供响应时限计算和多级升级通知，要在 ITSM 系统里自行配置。
- **用户级订阅过滤**：首版所有具备 ADMIN 或 OPERATOR 角色的用户都接收全量报警，不支持按设备或区域订阅。
- **WebSocket / SSE 实时推送**：站内通知基于 30 秒轮询实现，不提供长连接实时推送，极端场景下通知延迟最长约 30 秒。
- **多实例分布式部署**：首版为单实例运行，还不支持水平扩展的多实例集群部署。
- **自动化压力测试基线**：首版不内置压测基准或容量评估工具。
- **阈值配置热更新**：改超时阈值或失败次数阈值后，变更在下一个检测调度周期生效，最长延迟约 60 秒，不支持即时热更新。

## §6 与其他模块的关系

ems-alarm 在平台里是"质量哨兵"：它消费采集层的状态数据，结合设备元数据判断是否需要报警，把操作记录同步到审计模块；权限模块负责保证只有授权角色才能操作报警。采集数据质量的提升，间接让计费（ems-billing）和成本分析（ems-cost）模块拿到更完整的数据基础。ems-alarm 的 alarms 数据表对外封闭，其他模块只能通过 ems-alarm 提供的接口读取报警状态，不允许直接写入。

```
ems-collector ──采集状态（consecutiveErrors）──┐
                                              ├──> ems-alarm ──站内通知──> 铃铛 / 通知抽屉
ems-meter ────设备元数据（阈值、维护模式）────┘          │
                                                        ├──> Webhook ──> 企业 IM / ITSM
                                                        └──> 审计日志（ems-audit）

ems-auth ────角色权限校验──> ems-alarm（操作鉴权）
ems-billing / ems-cost ────间接受益于数据完整性
```

## §7 同类竞品对比（可选，简短）

| 对比维度 | ems-alarm | 开源 Zabbix / Prometheus | 简单 Webhook 通知脚本 |
|---|---|---|---|
| 部署复杂度 | 零额外部署，原生集成 EMS 平台 | 需独立服务器、配置、维护 | 轻量，但无状态管理 |
| 业务语义 | 设备级阈值覆盖、维护模式带审计，贴近能源管理业务 | 通用监控，需大量自定义 | 无业务感知 |
| 报警闭环 | 状态机（触发→确认→解决），双向恢复 | 有报警状态，需额外配置 | 无状态，无闭环 |
| 推送安全 | HMAC-SHA256（消息签名标准）+ 失败重试 + 下发流水 | 支持多种认证方式 | 通常无签名验证 |
| 上手成本 | 配置一个地址即用，界面操作友好 | 学习曲线陡，运维负担重 | 极低，但能力有限 |

**核心差异化**：ems-alarm 不用独立运维报警基础设施，与设备管理、权限、审计模块天然打通，业务语义（维护模式、设备级阈值）开箱即用，是能源管理场景下性价比最优的报警方案。
