# 采集中断告警 · 功能概览

> 适用版本：v1.6.0+ ｜ 最近更新：2026-04-29

---

## §1 一句话价值

实时监测每台设备的数据采集状态，断了第一时间通知，避免数据缺失影响计费、报表、能效分析。
无需人工盯盘，告警到手、处理有记录、恢复有闭环。

---

## §2 解决什么问题

- **仪表掉线无感知，损失数据才发现**
  仪表断电、网络中断后，系统不报警、运维不知情，等月底对账时才发现数据缺失，补录成本高、精度差。

- **采集程序卡死或网络抖动，KPI 报表出现数据断点**
  采集进程假死或网络短暂抖动时，曲线数据出现空洞，能耗分析、高峰核查等 KPI 报表失真，影响管理决策。

- **维护期间告警泛滥，运维团队产生"狼来了"麻木**
  设备计划停机或例行维护时，告警持续轰炸，团队逐渐忽视通知，真正异常来临时反而错过响应窗口。

- **多套上游系统需要同步通知，信息孤岛难打通**
  运维团队用钉钉，IT 部门用企业微信，工单系统又是另一套，每次异常需要手动广播，容易遗漏。

- **不同设备业务重要度不同，统一阈值不够用**
  关键主变压器和普通照明回路对断线容忍度差异巨大，一刀切的超时阈值要么漏报要么误报。

## §3 核心功能

- **双口径检测**：支持"静默超时"（多久没收到数据）和"连续失败"（连续多少次采集错误）两种检测口径，两者满足其一即触发告警，覆盖更多断线场景。
- **状态机闭环**：告警经历"触发 → 已确认 → 已解决"完整流程，支持系统自动恢复和人工手动关闭两种方式，每次操作留审计记录。
- **设备级阈值覆盖 + 维护模式**：可对单台设备单独配置超时阈值，启用维护模式期间自动屏蔽告警，操作时须填写备注，全程可审计。
- **站内通知**：管理员和运维人员登录后可见铃铛角标和通知抽屉，系统每 30 秒轮询一次，无需额外客户端。
- **Webhook 外发**：告警触发时主动推送到企业自定义地址，载荷使用 HMAC-SHA256（消息签名标准）确保安全，发送失败自动重试并记录下发流水，便于核查。
- **健康总览看板**：一屏展示全场设备的采集健康状态，内置"近 7 天告警最多 Top10 设备"排行，快速定位高频异常源。

## §4 适用场景

### 场景 A：能源管理项目方 — 保障报表数据完整性

能源审计或碳核查项目通常对数据完整率有硬性要求（如不低于 95%）。部署采集中断告警后，仪表断线 5 分钟内即可收到通知，运维人员快速响应、补救采集窗口，避免月底对账时发现大段数据缺失。对于重要度高的主回路，可单独调低超时阈值，做到"漏一个点都不放过"；对于次要回路，则适当放宽，减少噪音。

### 场景 B：工厂自动化运维团队 — 区分维护期与真实异常

大型工厂有定期停机计划，逢节假日或设备年检，数十台仪表会同时离线。运维负责人可在停机前为相关设备开启"维护模式"，告警自动暂停，维护结束后关闭即可恢复监测。真正的非计划断线依然正常报警，团队不再因维护期间的噪音告警而降低对真实故障的敏感度。

### 场景 C：多系统集成 — Webhook 桥接，不另开通道

企业内已有钉钉机器人、企业微信群或 ITSM 工单平台的，可直接将 Webhook 地址配置为对应系统的入口。告警触发时，ems-alarm 携带签名自动推送，接收方无需改造，一个 Webhook 地址即可将告警同步到现有通知渠道，无需为能源管理再单独部署告警中间件。

## §5 不在范围（首版）

以下能力在首个发布版本（v1.6.0）中尚不支持，已纳入后续路线图，购买前请与项目方确认：

- **邮件 / 短信 / IM 直连**：首版仅提供 Webhook 推送，钉钉、企业微信等 IM 接收方需通过 Webhook 桥接自行对接，不内置直连。
- **多个 Webhook 端点**：首版只支持配置一个全局 Webhook 地址，多目标分发需在接收方做转发。
- **告警严重程度分级**：首版所有告警统一为普通级别（NORMAL），不支持"紧急 / 警告 / 提示"分级，该能力规划在 v1.7+ 版本。
- **告警 SLA 计算与升级路由**：首版不提供响应时限计算和多级升级通知，需自行在 ITSM 系统中配置。
- **用户级订阅过滤**：首版所有具备管理员（ADMIN）或运维员（OPERATOR）角色的用户均接收全量告警，不支持按设备或区域订阅。
- **WebSocket / SSE 实时推送**：站内通知基于 30 秒轮询实现，不提供长连接实时推送，极端场景下通知延迟最长约 30 秒。
- **多实例分布式部署**：首版为单实例运行，尚不支持水平扩展的多实例集群部署。
- **自动化压力测试基线**：首版不内置压测基准或容量评估工具。
- **阈值配置热更新**：修改超时阈值或失败次数阈值后，变更在下一个检测调度周期生效，最长延迟约 60 秒，不支持即时热更新。

## §6 与其他模块的关系

ems-alarm 在整个平台中扮演"质量哨兵"角色：它消费采集层的状态数据，结合设备元数据判断是否需要告警，并将操作记录同步到审计模块，同时收益于权限模块保证只有授权角色才能操作告警。采集数据质量的提升，间接使计费（ems-billing）和成本分析（ems-cost）模块获得更完整的数据基础。ems-alarm 的 alarms 数据表对外封闭，其他模块只能通过 ems-alarm 提供的接口读取告警状态，不允许直接写入。

```
ems-collector ──采集状态（consecutiveErrors）──┐
                                              ├──> ems-alarm ──站内通知──> 铃铛 / 通知抽屉
ems-meter ────设备元数据（阈值、维护模式）────┘          │
                                                        ├──> Webhook ──> 企业 IM / ITSM
                                                        └──> 审计日志（ems-audit）

ems-auth ────角色权限校验──> ems-alarm（操作鉴权）
ems-billing / ems-cost ────间接受益于数据完整性
```

## §7 同类竞品对比（可选，简短）

| 对比维度 | ems-alarm | 开源 Zabbix / Prometheus | 简单 Webhook 通知脚本 |
|---|---|---|---|
| 部署复杂度 | 零额外部署，原生集成 EMS 平台 | 需独立服务器、配置、维护 | 轻量，但无状态管理 |
| 业务语义 | 设备级阈值覆盖、维护模式带审计，贴近能源管理业务 | 通用监控，需大量自定义 | 无业务感知 |
| 告警闭环 | 状态机（触发→确认→解决），双向恢复 | 有告警状态，需额外配置 | 无状态，无闭环 |
| 推送安全 | HMAC-SHA256（消息签名标准）+ 失败重试 + 下发流水 | 支持多种认证方式 | 通常无签名验证 |
| 上手成本 | 配置一个地址即用，界面操作友好 | 学习曲线陡，运维负担重 | 极低，但能力有限 |

**核心差异化**：ems-alarm 无需独立运维告警基础设施，与设备管理、权限、审计模块天然打通，业务语义（维护模式、设备级阈值）开箱即用，是能源管理场景下性价比最优的告警方案。
